# 阶段1：问题矩阵

## 1. 总览

本文件用于把当前文本系统中已经暴露出来的结构性问题分组整理，便于后续阶段按优先级推进。

分组维度：
- 内容模型
- 文本框布局
- 编辑器
- 渲染
- 复制粘贴
- structured import
- 迁移与扩展风险

---

## 2. 问题矩阵

| 编号 | 类别 | 问题 | 当前表现 | 根因判断 | 优先级 |
|---|---|---|---|---|---|
| P1 | 内容模型 | 文本内容没有唯一主源 | `html / plainText / text / editor DOM` 同时参与事实判断 | 缺少正式 `RichTextDocument` | 高 |
| P2 | 内容模型 | `html` 既是编辑缓存又是显示来源 | 编辑态与显示态依赖同一 HTML 快照 | HTML 被当作主模型使用 | 高 |
| P3 | 内容模型 | `plainText` 不是单纯派生字段 | 搜索、标题、尺寸、复制都会反向依赖它 | 派生层与主源层未分离 | 高 |
| P4 | 文本框布局 | 文本框模式不完整 | 只有 `auto-width / wrap`，没有正式 `fixed-size` | 缺少文本框布局协议 | 高 |
| P5 | 文本框布局 | `wrapMode` 与 `textResizeMode` 语义缠绕 | 行为来自两个字段共同决定 | 历史迭代叠加 | 高 |
| P6 | 文本框布局 | 文本框收敛规则不统一 | 编辑提交后框体何时收缩、何时固定不清 | 缺少布局状态机 | 高 |
| P7 | 测量 | 文本尺寸判定来源过多 | editor DOM、hidden host、canvas、heuristic 混用 | 有测量层但无统一主规则 | 高 |
| P8 | 测量 | fallback 仍依赖字符长度估算 | 混排、不同字体、列表时误差容易放大 | 未完全由结构化布局驱动 | 中 |
| P9 | 编辑器 | `contenteditable` 仍承担内容事实角色 | 编辑结果直接以 HTML 回写元素 | 编辑器桥接层未独立 | 高 |
| P10 | 编辑器 | 当前编辑器命令体系扩展成本高 | 列表、链接、复杂样式、未来公式都会继续变复杂 | 命令式 DOM 编辑方式已接近上限 | 中 |
| P11 | 渲染 | 编辑态与显示态不完全同构 | 编辑器里能看到的内容，不一定在 canvas 上完全一致 | 显示态靠 HTML 反解析 | 高 |
| P12 | 渲染 | 富文本支持范围取决于 HTML 解析器 | 局部样式、列表、链接等边界行为不稳定 | 没有统一文档模型到渲染 runs 的映射 | 中 |
| P13 | 复制粘贴 | clipboard broker 服务的是“元素集合”，不是“文本文档” | 复制时输出 `plainText/html`，但没有统一文本 clipboard 协议 | 复制体系以画布元素为中心 | 高 |
| P14 | 复制粘贴 | 画布态粘贴与编辑态粘贴是两套系统 | 画布态走 structured import，编辑态走 `contenteditable` | paste 路由未统一 | 高 |
| P15 | 复制粘贴 | 外部粘贴内容与内部编辑器模型没有正式桥接 | 粘贴进入画布与进入文本编辑态结果不一致 | 缺少 `clipboard -> RichTextDocument` 标准化层 | 高 |
| P16 | structured import | 文本导入系统输出的是画布元素，不是编辑器共享主模型 | parser / renderer 已做结构化，但结果仍落到 `html/plainText` | structured import 与 editor 体系未打通 | 高 |
| P17 | structured import | Markdown / code / math 等文本型输入与富文本编辑未共享协议 | 导入是一套逻辑，编辑又是一套逻辑 | 缺少共同文本主模型 | 中 |
| P18 | 兼容 | 旧文本数据迁移缺少正式制度 | 目前主要靠 normalize 推断 | 没有文本系统迁移协议 | 中 |
| P19 | 扩展 | 继续叠加功能风险高 | 再加 Markdown / LaTeX / 列表 / 协同会越来越难维护 | 基础架构还未完成分层 | 高 |
| P20 | 测试 | 缺少文本系统专项回归基线 | 现在更多是功能回归，不是文本架构回归 | 还没有正式文本系统测试矩阵 | 中 |

## 3. 当前最关键的结构性断点

### 3.1 断点一：内容主源断裂

当前文本系统最根本的问题不是某一个 bug，而是：

- 编辑器的主源是 editor DOM
- 元素持久化主源近似是 `html`
- 画布搜索与复制又强依赖 `plainText`

这会导致：
- 字段之间反复同步
- 行为很难完全一致
- 后续扩展越来越依赖 patch

结论：
- 必须引入唯一文本主源

---

### 3.2 断点二：文本框模式缺失

当前文本框不是正式状态机，而是字段推断。

问题表现：
- 单击创建和拖拽创建没有形成稳定语义差
- 手动改框后没有完整的固定框模式
- 收敛行为难以预测

结论：
- 必须建立正式的 `TextBoxLayoutMode`

---

### 3.3 断点三：clipboard 与 editor 体系断裂

当前复制粘贴与富文本编辑器没有完全接通。

问题表现：
- 画布态粘贴：structured import
- 编辑态粘贴：浏览器 / adapter
- 两边不共享统一的文本内容协议

结论：
- 必须建立 `TextClipboardProtocol`

---

## 4. 优先级排序

### 第一优先级

1. 唯一文本主源
2. 文本框三模式
3. 测量协议统一
4. clipboard 与 editor 打通

### 第二优先级

1. structured import 到文本主源桥接
2. 显示态渲染从 HTML 反解析迁移到文档模型驱动
3. 旧数据迁移策略正式化

### 第三优先级

1. 高级富文本能力扩展
2. Markdown / LaTeX 深度整合
3. 协同编辑支持
4. 高质量 PDF / 导出一致性

---

## 5. 阶段1落实结论

现阶段不应继续在旧架构上直接堆更多文本功能。

后续文本重构最合理的推进顺序应当是：

1. 先整理文本主源
2. 再整理文本框布局模式
3. 再整理测量协议
4. 再把 clipboard 与 editor 正式打通
5. 最后才是高级富文本能力扩展
