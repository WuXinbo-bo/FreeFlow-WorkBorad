# 阶段1：现状架构说明

## 1. 目标

本文件用于冻结当前 FreeFlow 画布文本系统、富文本编辑系统、复制粘贴系统与结构化导入系统的现状实现，为后续文本重构提供统一基线。

本阶段不修改逻辑，仅回答以下问题：

- 当前文本元素是如何创建、编辑、渲染、提交和持久化的
- 当前复制 / 粘贴 / 拖拽输入链路如何进入画布
- 当前富文本编辑器与复制粘贴体系的边界在哪里
- 当前有哪些真实的职责重叠与断层

---

## 2. 核心模块与职责

### 2.1 文本元素模型

文件：
- [text.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js)

当前职责：
- 规范化文本元素基础字段
- 维护文本最小尺寸与初始尺寸
- 推断文本尺寸模式
- 在 `html / plainText / text` 三组字段之间做归一化

当前关键字段：
- `text`
- `plainText`
- `html`
- `wrapMode`
- `textResizeMode`
- `width`
- `height`
- `fontSize`
- `structuredImport`

现状判断：
- `html`、`plainText`、`text` 都在参与内容状态维护
- `wrapMode` 与 `textResizeMode` 共同影响尺寸行为
- 当前没有明确的独立富文本文档主模型

---

### 2.2 文本测量系统

文件：
- [measureTextElementLayout.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/textLayout/measureTextElementLayout.js)

当前职责：
- 统一测量文本元素内容尺寸
- 支持编辑态 DOM 测量
- 支持隐藏 host DOM 测量
- 支持 canvas fallback 测量
- 最后再退回字符数启发式估算

当前测量优先级：
1. `editorElement`
2. 隐藏测量 host
3. `canvas` runs 测量
4. 纯文本估算

输入：
- `html`
- `plainText`
- `fontSize`
- `resizeMode`
- `widthHint`
- `maxWidth`
- `editorElement`

输出：
- `contentWidth`
- `contentHeight`
- `frameWidth`
- `frameHeight`
- `measuredWith`
- `reason`

现状判断：
- 这是当前文本系统里最接近正式协议的一层
- 但其输入主源仍然依赖 `html/plainText`，不是结构化文档模型

---

### 2.3 富文本编辑器适配层

文件：
- [richTextAdapter.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/editors/richTextAdapter.js)

当前职责：
- 管理 `contenteditable` 宿主
- 管理选区、提交、取消
- 执行加粗、斜体、下划线、删除线、列表、对齐、字号、颜色、高亮等命令
- 读取当前格式状态
- 输出当前 HTML / 纯文本片段

当前实现特征：
- 以 `contenteditable` 为基础
- 仍然依赖命令式编辑流程
- 内部带有大量 DOM 祖先查找、wrapper 识别、选区恢复逻辑

现状判断：
- 当前编辑器是“编辑层 UI”，但也部分承担了内容结构事实来源的角色
- 长期继续扩展，复杂度会快速上升

---

### 2.4 文本显示态渲染系统

文件：
- [rendererText.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js)

当前职责：
- 在 canvas 上绘制文本元素
- 解析当前 `html` 为富文本 runs
- 绘制局部样式，如：
  - bold
  - italic
  - underline
  - strike
  - highlight
  - color
  - inline font size

当前关键函数：
- `parseRichTextRuns`
- `layoutRichRuns`
- `drawRichTextInBox`
- `measureRichTextBox`
- `drawTextElement`

现状判断：
- 当前显示态是“HTML -> runs -> canvas draw”
- 编辑态与显示态并不共享同一份结构化内容模型
- 富文本支持范围取决于 HTML 解析器能识别的标签和 style

---

### 2.5 画布运行时主链路

文件：
- [createCanvas2DEngine.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js)

当前职责：
- 串联文本创建、编辑、提交、渲染刷新、历史记录、复制粘贴、导入导出
- 维护当前编辑态 ID 与类型
- 维护 `richEditor` DOM 宿主
- 维护工具条状态

当前文本编辑主链路：
1. 进入编辑：
   - `beginTextEdit`
   - `beginFlowNodeEdit`
2. 编辑过程中：
   - `syncActiveRichEditingItemState`
   - `syncEditingTextItemSize`
   - `syncEditingFlowNodeState`
3. 提交：
   - `commitTextEdit`
   - `commitFlowNodeEdit`
4. 取消：
   - `cancelFlowNodeEdit`
   - `cancelRichEdit`

当前关键事实：
- 文本编辑提交时仍然先从 editor 取 HTML
- 再由 `htmlToPlainText` 派生 `plainText`
- 再回写元素的 `html/plainText/text`
- 然后再用尺寸测量系统更新 `width/height`

现状判断：
- 运行时层承担了大量“字段同步”和“状态回写”职责
- 文本系统现在不是分层式架构，而是运行时调度驱动

## 3. 当前复制粘贴与结构化导入链路

### 3.1 内部复制 Broker

文件：
- [clipboardBroker.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/brokers/clipboardBroker.js)

当前职责：
- 从选中元素构造内部复制 payload
- 输出：
  - `items`
  - `text`
  - `html`
  - `filePaths`
- 向系统剪贴板尝试写入：
  - 文件
  - 内部 payload
  - `text/plain`
  - `text/html`

当前内部 payload 特征：
- 类型为 `canvas2d`
- 本质是 FreeFlow 自定义内部格式
- 对文本元素优先使用 `plainText / html`

现状判断：
- 当前 broker 是“元素复制 broker”，不是“富文本 clipboard protocol”
- 它服务于画布整体复制，不是专门服务于文本系统

---

### 3.2 粘贴入口 Gateway

文件：
- [pasteGateway.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/import/gateway/pasteGateway.js)
- [sharedEntryBuilders.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/import/gateway/sharedEntryBuilders.js)
- [contentTypeDetector.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/import/gateway/contentTypeDetector.js)

当前职责：
- 读取浏览器 `clipboardData`
- 读取系统剪贴板快照
- 将剪贴板内容标准化为 `Input Descriptor`
- 为后续 parser / renderer / adapter 链路提供统一输入

当前支持内容：
- `text/plain`
- `text/html`
- `text/markdown`
- `text/uri-list`
- 内部 payload
- 文件

当前设计特征：
- 入口层只负责描述符标准化
- 后续是否落为文本、列表、表格、代码、数学，由 parser / renderer 决定

现状判断：
- 这是“结构化导入系统的入口”
- 但它和富文本编辑器内部的 paste 行为不是同一系统

---

### 3.3 拖拽入口 Gateway

文件：
- [dragGateway.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/import/gateway/dragGateway.js)

当前职责：
- 将 `DataTransfer` 标准化为 descriptor
- 支持文件拖拽与文本拖拽的统一接入

现状判断：
- 拖拽系统与粘贴系统已在 descriptor 层统一
- 但仍未与富文本编辑器的内部插入模型统一

---

### 3.4 Structured Import Runtime

文件：
- [createStructuredImportRuntime.js](D:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/import/runtime/createStructuredImportRuntime.js)

当前职责：
- 组装：
  - paste gateway
  - drag gateway
  - parser registry
  - renderer pipeline
  - legacy adapter registry
  - commit layer
- 将外部输入一路转成画布元素

当前 parser 注册顺序：
- internal compatibility
- file resource
- image resource
- latex math
- code
- markdown
- web content
- html
- plain text

当前 renderer 注册顺序：
- generic text
- list
- code block
- table
- math
- image

现状判断：
- 新模式结构化导入已经是一套独立架构
- 但它输出的是“画布元素”
- 不是文本编辑器共享的正式富文本文档模型

## 4. 当前文本主链路全景

### 4.1 创建文本

当前流程：
- 用户创建文本元素
- `createTextElement` 生成：
  - `html`
  - `plainText`
  - `text`
  - `textResizeMode`
  - `width/height`

特点：
- 初始就依赖 `html + plainText`
- 没有独立 `RichTextDocument`

---

### 4.2 进入编辑

当前流程：
- `beginTextEdit`
- 初始化 `richTextAdapter`
- 将当前 `item.html` 填入 editor
- 同步当前字体、工具条和编辑框位置

特点：
- 编辑器输入主源是元素内现存 `html`

---

### 4.3 编辑过程中

当前流程：
- editor DOM 持续变化
- `syncEditingTextItemSize` 会读 editor HTML
- 再转 plain text
- 再测量
- 再回写元素尺寸

特点：
- 编辑过程中的“内容事实”在 editor DOM
- 元素字段则被持续回写

---

### 4.4 提交

当前流程：
- 读取 editor HTML
- 生成 `plainText`
- 若为空则删除元素
- 否则回写：
  - `html`
  - `plainText`
  - `text`
  - `title`
  - `width/height`

特点：
- 提交时仍然依赖 `HTML -> PlainText` 的双字段同步

---

### 4.5 显示渲染

当前流程：
- 非编辑态时读取 `item.html`
- 用 `rendererText.js` 将 HTML 解析成 runs
- 再绘制到 canvas

特点：
- 显示态不是直接读编辑器模型
- 而是读元素缓存下来的 HTML

---

### 4.6 复制

当前流程：
- 从元素集合构建 payload
- 文本元素复制时导出：
  - `plainText`
  - `html`
- 再尝试写系统剪贴板

特点：
- 复制系统的服务对象是“画布元素”
- 不是“文本编辑器文档”

---

### 4.7 粘贴

当前流程有两条：

1. 画布态粘贴
- 进入 paste gateway
- 进入 structured import
- 解析成画布元素

2. 文本编辑态粘贴
- 进入 `contenteditable`
- 由 rich text adapter / 浏览器默认行为处理

特点：
- 两条体系当前没有统一到一个共享文本模型

---

## 5. 当前架构结论

当前系统已经具备：
- 基础富文本编辑能力
- 文本尺寸测量系统
- 文本显示态渲染
- 结构化导入系统
- 复制 / 粘贴 / 拖拽基础框架

但当前系统仍然不是一套统一的“文本系统”，而是：

- 一套以 `html + plainText + text` 为核心的文本元素机制
- 一套以 `contenteditable` 为核心的富文本编辑机制
- 一套以 descriptor / parser / renderer 为核心的结构化导入机制
- 一套以元素复制为核心的 clipboard broker

这些机制之间已有连接，但还没有统一主源与统一协议。

---

## 6. 阶段1结论

阶段1冻结的事实如下：

1. 当前文本系统的真实主源并不明确
- `html`
- `plainText`
- `text`
- editor DOM
都在承担部分事实来源

2. 当前文本框行为依赖混合字段
- `wrapMode`
- `textResizeMode`

3. 当前复制粘贴系统与富文本编辑器系统未完全打通
- 画布态粘贴走 structured import
- 编辑态粘贴走 `contenteditable`

4. 当前显示态与编辑态未共享正式的结构化文档模型

5. 当前最适合推进的方向不是继续堆功能，而是：
- 先建立统一的文本内容主源
- 再建立统一的文本框布局模式
- 再建立 clipboard 与 editor 的统一协议
