# FreeFlow v1.0.0-rc 阶段 1：版本基线说明

## 1. 阶段目标

- 冻结当前可用代码基线，避免后续版本迭代过程中目标、命名和交付物分散。
- 明确当前项目状态、当前已知问题、兼容现状与下一阶段发布目标。
- 在后续阶段统一使用 `v1.0.0-rc` 作为当前唯一正式发布标识。

## 2. 当前版本状态

### 2.1 当前源码版本

- `package.json` 当前版本仍为：`0.8.0-beta`
- `productName` 当前为：`FreeFlow`
- 当前 Electron 打包产物命名仍跟随 `package.json.version`

### 2.2 当前版本日志状态

- 当前版本日志文件已切换为：
  - `docs/version-logs/v1.0.0-rc改动日志.md`
- 当前版本日志顶部标识已统一为：
  - `v1.0.0-rc`

### 2.3 当前 release 保留产物

- 当前 `release/` 中仅保留一个旧安装包：
  - `FreeFlow-0.8.0-beta-x64.exe`

### 2.4 当前 Git 状态

- 当前目录未检测到 `.git`
- 因此暂时无法建立 `v1.0.0-re` 或其他发布工作分支
- 若后续需要分支冻结发布流程，需先确认项目仓库已初始化或当前目录切换到真正的 Git 工作区

## 3. 当前升级链路与兼容状态

### 3.1 已确认较稳定的部分

- 用户数据目录延续策略已固定：
  - `%USERPROFILE%\\FreeFlow\\AppData`
  - `%USERPROFILE%\\FreeFlow\\CanvasBoards`
- 桌面应用标识当前保持稳定：
  - `appId = com.wuxinbo.freeflow`
- 启动链路中已存在最近画布 / 教程画布的专项修正逻辑
- 多数后端持久化存储使用版本化读写和 normalize 机制
- 当前代码已包含旧版画布元素到新画布元素结构的兼容映射

### 3.2 当前兼容结论

- 以当前代码结构来看，`v0.8.0-beta` 用户数据延续基础是成立的
- 当前代码已具备向 `v1.0.0-rc` 过渡的兼容底座
- 但当前项目还没有真正完成一次“版本号、安装包、升级路径、发布说明”全链路统一的正式发布状态

### 3.3 当前兼容风险点

- `package.json` 版本仍停留在 `0.8.0-beta`
- `release/` 中仍是旧版安装包命名
- 现有升级验证脚本主要覆盖后端存储链路，尚不足以单独作为“完整无缝升级”的最终证明
- 画布内部复制 / 外部复制 / 保留格式粘贴链路刚做过多轮修复，仍属于发布前重点验证项

## 4. 当前已知问题清单

### 高优先级

- 画布内部多选复制后，保留格式粘贴链路曾出现退化为普通文本粘贴的问题
- 画布内部复制与程序外复制之间的优先级切换逻辑刚重构，仍需发布前专项回归
- 当前项目不是 Git 工作区，无法进行分支冻结管理
- 打包版本号、日志版本号、安装包文件名目前仍未完全统一

### 中优先级

- 当前版本日志已使用 `v1.0.0-rc`，但 `package.json` 和 `release` 目录仍保留 `0.8.0-beta`
- `README.md` 仍不是面向 GitHub 发布的正式版本说明文档
- 旧版升级说明、发布说明、验收清单等发布资料尚未建立

### 发布前重点复核项

- 最近画布 / 教程画布首次引导逻辑
- 多选复制粘贴排版保留
- 外部复制后画布粘贴优先级切换
- 画布内全局搜索
- 富文本大字号与编辑工具条适配
- 主面板自由布局、全局浮层与桌面窗口 shape 稳定性

## 5. 版本命名冻结结论

### 当前冻结决定

- 本轮发布准备阶段仅保留一个正式发布标识：
  - `v1.0.0-rc`

### 本阶段不再并行使用的标识

- `v1.0.0-re`
- `v0.8.1`
- 其他混合表达（如日志、包名、README、发布说明中混用 `rc / re / beta`）

### 后续统一要求

- 版本日志统一使用：`v1.0.0-rc`
- 发布说明统一使用：`v1.0.0-rc`
- README 中当前目标版本统一使用：`v1.0.0-rc`
- 后续打包时，安装包名称也应统一切换到 `v1.0.0-rc`
