# FreeFlow v1.0.1-rc 结构化导入与复制粘贴架构改造实施版

## 1. 文档定位

本文件不再只是“能力清单”，而是面向实施的重置版方案。

本版目标：

- 基于当前项目真实代码现状，重新定义问题边界
- 明确为什么现有复制 / 粘贴 / 拖拽链路不足以支撑后续目标
- 给出与当前版本的详细对比
- 按任务包拆分改造范围、依赖关系、阶段目标与验收标准
- 为后续开发排期、测试、灰度切换提供统一依据

本次实施还必须遵守一个总前提：

- 后续这套改动按“完全独立的新系统”准备和实施
- 默认不侵入、不替换当前已有主链
- 新系统单独建目录、单独建入口、单独建协议和任务线
- 只有在必须共用现有能力时，才通过适配层桥接
- 在新系统完整做完、跑通、可回归、可灰度之前，不直接替换当前系统
- 最后再做“新旧切换 / 替换 / 回退”收口

这意味着：

- 接下来不是在旧链路上缝补
- 而是并行重做一套独立系统

本文件默认约束：

- 当前版本先作为兼容层保留，不直接推翻
- 当前阶段不纳入 AI 后处理、AI 自动整理、AI 语义增强
- 当前阶段重点只做“结构化导入、统一协议、原生渲染、元素升级、复制粘贴主链收口”

### 1.1 本文如何使用

本文件建议按以下方式使用：

- 产品 / 架构讨论阶段：先看第 2 到第 7 节，统一问题定义和目标
- 开发排期阶段：重点看第 8 到第 11 节，按任务包和阶段拆排期
- 测试与验收阶段：重点看第 11 到第 12 节，对照验收口径和风险点

### 1.2 本文不解决的问题

本文件不会解决以下问题：

- AI 自动整理与摘要策略
- 导出为 PDF / Word / Markdown 的反向协议设计
- 多人协同协议
- 画布整体布局和视觉设计系统重构

这些内容未来可以建立独立文档，不应混入当前主线。

## 2. 背景重置

### 2.1 当前项目遇到的根问题

当前 FreeFlow 已经具备以下可用能力：

- 画布内部元素复制
- 外部纯文本粘贴
- 外部 HTML 粘贴
- 文件拖拽导入
- 富文本编辑态内的局部复制 / 粘贴

但这些能力是“能用”的，并不是“统一的”。

现状的核心问题不是某一个 bug，而是整体架构仍停留在：

- 内部复制走原生元素克隆
- 外部导入走 MIME 粗分类
- HTML 内容多数直接落成文本元素
- 渲染层没有统一的结构化中间协议

这会直接导致：

- 新来源越多，入口特判越多
- HTML / Markdown / Office / 代码 / 数学公式等内容容易在导入后退化
- Ctrl + V、右键粘贴、拖拽导入之间能力不一致
- 后续想做表格、任务列表、代码块、注脚、公式时，只能继续往旧文本模型里硬塞

### 2.2 本次重置的目标

本次不是继续修补旧链路，而是把整体导入系统升级为：

`输入层 -> 解析层 -> canonical JSON -> 渲染层 -> FreeFlow 原生元素`

这次重置要解决的不是“把东西贴进来”，而是：

- 理解外部内容结构
- 用统一协议表达结构
- 用 FreeFlow 自己的元素体系承接结构
- 让内部复制与外部导入逐步走向同一主链

补充目标约束：

- 新体系优先独立搭建，不直接侵入当前线上主链
- 所有新模块默认以“并行系统”方式存在
- 当前系统在新体系完成前继续作为稳定兼容层存在

### 2.3 为什么现在必须做这件事

如果继续沿用当前模式，后续每新增一种来源都会带来三类连锁问题：

1. 入口层继续分叉  
   例如：Markdown 一套、网页 HTML 一套、Office 一套、公式一套。

2. 文本元素继续膨胀  
   所有复杂结构最终都塞进 `text/html/plainText`，编辑、复制、命中、导出都会越来越脆。

3. 行为越来越不一致  
   同一份内容从 Ctrl + V、右键菜单、拖拽、编辑器内粘贴进入，结果会继续分化。

所以本次不是“提前设计”，而是到了必须收敛的时候。

## 3. 当前代码基线

本节只记录当前项目与本次改造最相关的真实代码基线，后续所有任务包都以这里为基础展开。

### 3.1 当前核心代码位置

| 模块 | 当前文件 | 当前职责 | 当前局限 |
| --- | --- | --- | --- |
| 内部复制 broker | `public/src/engines/canvas2d-core/brokers/clipboardBroker.js` | 把选中元素打包为内部 payload，并写入文本 / HTML / 文件路径 | payload 本质是原生 item 克隆，不是统一结构协议 |
| 外部拖拽 / 粘贴 broker | `public/src/engines/canvas2d-core/brokers/dragBroker.js` | 按 `files / html / text` 分流导入 | HTML 会直接落成一个文本元素，没有结构化解析 |
| 主链入口 | `public/src/engines/canvas2d-core/createCanvas2DEngine.js` | 组合 clipboard、drag、编辑器、右键菜单、粘贴行为 | Ctrl + V、右键粘贴、拖拽导入未收口到同一协议 |
| 富文本编辑器适配层 | `public/src/engines/canvas2d-core/editors/richTextAdapter.js` | 负责编辑态 HTML、选区样式、编辑器内复制粘贴 | 主要服务编辑器内部，不是统一导入协议 |
| 文本元素协议 | `public/src/engines/canvas2d-core/elements/text.js` | 生成与归一化文本元素 | 当前主要承接 `text/html/plainText`，不是结构化文档块 |
| flow 文本元素协议 | `public/src/engines/canvas2d-core/elements/flow.js` | 节点类文本内容承接 | 同样主要依赖 HTML / plainText |
| HTML / 文本基础工具 | `public/src/engines/canvas2d-core/utils.js` | `sanitizeHtml`、`htmlToPlainText`、`normalizeRichHtmlInlineFontSizes` 等 | 更偏轻量清洗，不是多来源结构解析器 |

### 3.2 当前代码链路结论

#### 3.2.1 内部复制

当前内部复制由 `clipboardBroker.buildPayloadFromItems()` 构建 payload。

其核心内容是：

- `items`: 原生画布元素克隆
- `text`: 提取的纯文本摘要
- `html`: 仅对文本元素拼接 HTML
- `filePaths`: 文件 / 图片路径

这说明当前项目的“内部中间层”实际上是原生 item，不是统一文档协议。

#### 3.2.2 外部导入

当前 `dragBroker.importFromDataTransfer()` 逻辑是：

1. 先看 `files`
2. 再看 `text/html`
3. 最后看 `text/plain`

其中 HTML 导入现在会：

- 调 `normalizeRichHtmlInlineFontSizes`
- 调 `htmlToPlainText`
- 最后 `createTextElement(point, plainText, html)`

这意味着外部 HTML 内容目前基本是“轻量清洗后直接塞进文本元素”，不是“解析为结构树再渲染”。

#### 3.2.3 编辑器内部粘贴

`richTextAdapter.js` 里有独立的编辑器内复制粘贴逻辑。

它会：

- 读取 `text/html` / `text/plain`
- 归一化 HTML
- 直接插入到 contenteditable

这条链和画布外部导入链不是同一协议。

#### 3.2.4 当前元素模型

`text.js` 当前核心字段是：

- `text`
- `html`
- `plainText`
- `fontSize`
- `width`
- `height`

这对于轻量富文本足够，但对于：

- 任务列表
- 表格
- 代码块
- 数学公式
- 注脚

并不是天然合适的数据结构。

### 3.3 当前项目可直接复用的基础

- 当前 clipboard / drag / paste 入口
- 当前文本元素、flow 元素、图片元素、文件卡片
- 当前 HTML 清洗、字号归一、plain text 提取工具
- 当前 rich text 编辑器和工具条能力
- 当前桌面端剪贴板桥接能力

### 3.3.1 现有画布元素体系纳入范围建议

基于本次结构化导入升级，现有画布元素体系应分层纳入，而不是统一粗暴处理。

主线纳入对象：

- 文本元素
- 图片元素
- 文件卡片
- 列表 / 任务列表
- 表格
- 代码块
- 数学公式
- 引用块 / 注脚相关文档块

兼容观察对象：

- flow 节点
- 节点文本
- 图片备注
- 文件备注
- 现有组合型原生内容

暂不纳入结构化导入主线对象：

- 手绘笔迹
- 几何图形
- 连线 / 箭头
- 涂鸦 / 高亮笔
- 自由画图对象

建议原则：

- 主线纳入对象进入新体系主链
- 兼容观察对象进入样本、复制、回归与适配层观察
- 暂不纳入主线对象继续沿用现有原生体系，只纳入外围兼容观察

### 3.3.2 现有元素类型全量盘点与同类项合并结论

基于当前代码中的 `elements/index.js`，目前旧画布真实存在的元素类型可归纳为：

| 旧元素类型 | 代码别名 / 归一化入口 | 当前职责 | 与新体系的同类项 | 第一阶段结论 |
| --- | --- | --- | --- | --- |
| `text` | `richText` / `rich` 最终都归一到 `text` | 基础文本、富文本正文 | `paragraph / heading / blockquote / inline marks` | 已由现有 parser 主线覆盖，后续进入文本 renderer 与文本元素增强 |
| `image` | `normalizeImageElement()` | 图片资源、尺寸、批注、memo | `image` canonical 节点 | 属于直接同类项，必须补“图片资源 parser + image bridge renderer” |
| `fileCard` | `file` 最终也归一到 `fileCard` | 文件卡片、文件路径、memo | 无完整 canonical 同类节点 | 不应硬塞进文档 canonical，需补“文件资源 compatibility adapter + fileCard bridge renderer” |
| `flowNode` | `flow-node` / `node` | 节点文本容器 | 无直接 canonical 同类项 | 不属于第一批文档 parser 目标，应走后续旧原生元素兼容 adapter |
| `flowEdge` | `flow-edge` / `edge` | 节点之间连线关系 | 无直接 canonical 同类项 | 不属于第一批 parser 目标，应保留在旧原生关系层 |
| `mindNode` | `mind` | 思维节点块 | 无直接 canonical 同类项 | 不属于第一批 parser 目标，应走后续兼容 adapter |
| `shape` | `shapeType=rect/ellipse/line/arrow/highlight` | 几何图形、线、箭头、高亮 | 无直接 canonical 同类项 | 不属于第一批 parser 目标，应留在画布原生创作层 |

合并同类项后的结论：

1. 新体系当前已经覆盖的“同类项”主要是：
   - `text`
   - 文档结构块
   - `codeBlock`
   - `table`
   - `mathInline / mathBlock`

2. 与旧元素体系直接同类、但目前仍缺兼容闭环的只有两类：
   - `image`
   - `fileCard`

3. `flowNode / flowEdge / mindNode / shape` 不应误判为“结构化导入 parser 缺失项”。
   它们本质上属于画布原生元素，不应在第一批 parser 中硬做文档化，而应在后续 compatibility adapter 阶段处理。

### 3.3.3 对第一批 parser 的补充判断

基于上面的全量元素盘点，原先的第一批 parser 虽然已经完成了文本、HTML、Markdown、代码、公式这条文档主线，但若目标是“与现有旧元素体系兼容”，则还需要补三类兼容 parser / adapter：

1. 图片资源 Parser  
   作用：把 `INPUT_ENTRY_KINDS.IMAGE` 与 `image-resource` 来源直接映射为 canonical `image` 节点，避免长期只停留在 fallback。

2. 文件资源 / FileCard Compatibility Adapter  
   作用：把 `INPUT_ENTRY_KINDS.FILE` 与 `file-resource` 来源收敛成新系统可识别的兼容输出，再桥接回旧 `fileCard` 元素。

3. 内部旧元素 Compatibility Parser  
   作用：让旧系统内部复制出来的 native item payload 在进入新系统时能被分流识别，而不是只有 `text/html/filePath` 路径。

这三类补充并不是“继续扩展文档 parser 范围”，而是为了确保新系统后续接回旧主链时，不会把现有图片、文件卡片和内部旧元素链路漏掉。

### 3.4 当前项目必须重构的部分

- 入口层分流过重
- 内部复制协议与外部导入协议割裂
- HTML 直接落文本元素的主路径
- 结构缺失后只能依赖 `html/plainText`
- 右键粘贴弱于 Ctrl + V

### 3.5 当前代码基线对本次方案的直接影响

当前代码现状决定了本次实施必须遵守这几条约束：

- 不能一开始就删掉 `clipboardBroker` 和 `dragBroker`
- 不能一开始就把 `text.js` 彻底废弃
- 不能指望编辑器内部 `richTextAdapter` 直接升级成统一导入器
- 不能只改 schema，不改 renderer 与 element

换句话说，本次改造必须是“并行架构 + 渐进迁移”，不能是“推翻重做”。

### 3.6 画布体系额外注意事项

除了输入、解析、协议、渲染和元素本身，本次改造还必须同步观察整个画布体系中已经与元素模型强耦合的系统模块。

如果这些模块不在前期被纳入基线与设计范围，那么新系统即使独立完成，后续接回主链时也很容易出现“元素能生成，但交互、搜索、导出、撤销不工作”的结构性问题。

重点关注以下八类：

1. 命中与选中系统  
当前命中逻辑主要围绕 `text`、`shape`、`flowEdge`、`fileCard`、`image` 等既有元素实现。后续若接入表格单元格、代码块局部编辑区、公式编辑区、task list 勾选区，不能继续只依赖整体 bounds 命中。

2. 搜索系统  
当前搜索索引主要覆盖 `text`、`flowNode`、`fileCard`、`image`。后续新增 `codeBlock`、`table`、`math`、`taskList`、`footnote` 等后，如果不补搜索索引映射，新内容将对搜索系统不可见。

3. 导出链路  
当前导出准备逻辑对 `text` 元素存在专门处理。后续新元素必须定义自己的导出测量、降级与兼容输出规则，否则导出 PDF / 图片时会丢内容或失真。

4. 历史记录与撤销重做  
新结构化元素会带来更细粒度操作，如表格单元格编辑、任务勾选、公式源码切换、代码块语言切换。若历史记录仍按旧对象粒度运作，后续撤销重做行为将不稳定。

5. Store 与持久化状态边界  
新系统按要求应独立建设，因此必须提前定义哪些状态属于独立新系统，哪些状态通过适配层与旧画布共享，避免后续新旧状态彼此污染。

6. 渲染器注册机制  
当前 renderer 体系围绕旧元素组织。新系统不应继续直接往旧 renderer 内堆分支，而应建立独立的 structured renderer 注册与适配层，最后再与旧渲染入口桥接。

7. 工具体系  
表格编辑、代码块编辑、公式编辑、任务列表勾选等都不是纯 renderer 问题，还会影响选择工具、平移工具、创建工具与编辑态切换逻辑，因此必须纳入观察。

8. 附属编辑区与 memo 模式  
当前文件卡片与图片已存在主体元素与附属编辑区的模式。后续表格单元格、公式源码区、代码块头部、task checkbox 等都可能复用这种模式，但需要统一协议，避免继续散落成局部特判。

## 4. 外部规范与资料依据

本次方案不只基于项目代码，也参考了当前主流规范与实现资料，用于约束协议和避免重复踩坑。

### 4.1 参考规范

- CommonMark Spec（规范 Markdown 基础语法）
  - https://spec.commonmark.org/
- GitHub Flavored Markdown Spec（任务列表、表格、删除线、脚注等 GFM 扩展）
  - https://github.github.com/gfm/
- KaTeX Auto-render / Delimiters（LaTeX 公式常见分隔符与渲染约定）
  - https://katex.org/docs/autorender

### 4.2 参考实现方向

当前项目 `package.json` 中尚未引入 Markdown / 数学公式解析栈。

后续如进入实现，可优先评估：

- Markdown / GFM:
  - `unified`
  - `remark-parse`
  - `remark-gfm`
  - `remark-math`
- 数学公式渲染:
  - `katex`

选择理由：

- 这些工具更接近“源码 -> AST / 结构节点”的路线
- 比先转 HTML 再二次解析更符合本次架构目标
- 能更好支撑任务列表、注脚、表格、数学公式等结构

### 4.3 资料参考的使用边界

这些外部规范和实现资料的作用是：

- 约束协议设计
- 避免功能定义偏离主流生态
- 降低后续接第三方库时的适配成本

但它们不会直接决定 FreeFlow 的最终画布表达。  
例如：

- GFM 的“脚注”在 FreeFlow 中不一定照搬文档底部模型
- KaTeX 的渲染方式不等于画布中的最终公式元素交互

规范只约束“输入语义”和“中间协议”，不直接决定“画布交互语义”。

## 5. 当前版本与目标版本详细对比

| 维度 | 当前版本 | 目标版本 |
| --- | --- | --- |
| 输入入口 | 内部复制、原生粘贴、右键粘贴、拖拽各自分流 | Import Gateway 统一收口 |
| 来源识别 | 主要按 MIME 粗分 | 按来源与内容特征联合识别 |
| 解析方式 | 通用轻量清洗 | 专属 parser + 统一调度 |
| 中间层 | 原生 item + `text/html/plainText` 混用 | 全局唯一 canonical JSON |
| Markdown | 无原生协议 | 原生 Markdown / GFM 结构解析 |
| 数学公式 | 仅有编辑器层富文本能力，未形成导入协议 | `mathInline / mathBlock` 与多来源公式预留 |
| 渲染方式 | 主要落成文本元素 | 按结构节点生成原生块级元素 |
| 元素模型 | 文本元素承接过多内容 | 文本增强 + 新增代码块 / 表格 / 公式等元素 |
| 扩展方式 | 新来源容易加特判 | 新来源通过 parser / renderer 注册扩展 |
| 一致性 | Ctrl + V / 右键粘贴 / 拖拽不一致 | 最终进入统一导入主链 |
| 可维护性 | 随来源增多复杂度持续上升 | 协议化后长期扩展成本更低 |

## 6. 重置后的总目标

### 6.1 目标架构

建议收敛为五段主链：

`Input Gateway -> Parser Registry -> Canonical Document -> Renderer Pipeline -> Canvas Elements`

### 6.2 核心原则

- 不再把外部 HTML 视为最终协议
- 所有来源先归一成 canonical JSON
- renderer 只吃 canonical JSON
- canvas element 只承接结构化结果
- 内部复制与外部导入逐步共享同一协议层

### 6.3 扩展预留

- 自定义解析器接口
- 自定义渲染器接口
- MathML / OMML 公式来源预留
- 后续可插拔来源识别与质量诊断能力

### 6.4 本次重置后的主链职责边界

建议严格控制五段主链的职责：

| 层级 | 只负责什么 | 不负责什么 |
| --- | --- | --- |
| Input Gateway | 收集原始输入、规范入口、产出 Input Descriptor | 不直接解析结构，不直接创建画布元素 |
| Parser Registry | 选择解析器、把原始输入转为 canonical JSON | 不直接渲染，不直接命中文本元素 |
| Canonical Document | 统一表达结构与样式语义 | 不直接关心画布坐标、选区和 UI |
| Renderer Pipeline | 把结构节点映射为画布原生元素 | 不负责来源识别，不负责系统剪贴板读写 |
| Canvas Elements | 承载最终编辑、命中、拖拽、复制能力 | 不再反向承担来源解析职责 |

## 7. canonical JSON v1 重置要求

### 7.1 最低必须支持的结构

- 文档根节点
- 段落
- 标题
- 普通列表
- 任务列表
- 表格
- 代码块
- 行内代码

- 数学公式：`latex/mathml/omml/sourceFormat/displayMode`
- Markdown / GFM：`taskList/taskItem/footnote/footnoteRef`

### 7.3 输入描述对象最低字段

```js
{
  entry: "paste" | "context-menu-paste" | "drag" | "internal-copy",
  sourceKind: "internal" | "html" | "plain-text" | "markdown" | "office" | "web" | "code" | "math-formula" | "files",

  errorCode: "",

    html: "",
    text: "",
    markdown: "",
    mathml: "",
    officeFormula: "",
    files: []
  },
  meta: {
    timestamp: 0,
    fromApp: "",
    mimeTypes: [],
    sourceUrl: "",
    sourceFilePath: ""
  }
}
```

### 7.4 canonical JSON v1 最低节点枚举建议

建议第一版就明确节点枚举，不要只停留在口头描述。

块级节点建议至少包括：

- `doc`
- `paragraph`
- `heading`
- `blockquote`
- `list`
- `listItem`
- `taskList`
- `taskItem`
- `table`
- `tableRow`
- `tableCell`
- `codeBlock`
- `mathBlock`
- `image`
- `divider`
- `footnote`

行内节点建议至少包括：

- `text`
- `hardBreak`
- `link`
- `inlineCode`
- `mathInline`
- `footnoteRef`

### 7.5 canonical JSON v1 最低元数据要求

建议第一版就为文档和节点预留元数据位，避免后续二次破坏协议。

推荐预留：

- `version`
- `source.kind`
- `source.app`
- `source.mimeTypes`
- `source.url`
- `source.filePath`
- `diagnostics`
- `losses`
- `warnings`

### 7.6 第一版明确不纳入 canonical 的内容

为了控制范围，以下内容不建议进入 v1：

- 复杂排版坐标
- 任意 CSS 样式映射
- 绝对定位布局
- 浮动环绕
- 分页语义
- 文档级页眉页脚

这些能力与画布原生表达关系较弱，放进 v1 会显著拉高复杂度。

### 8.3 任务包并行原则

允许并行的前提：

- 写入面尽量不重叠
- 上下游协议已冻结
- 不并行修改同一条核心入口链

因此比较适合并行的是：

- TP-04 / TP-05 / TP-06 的 parser 开发
- TP-07 renderer 骨架与 TP-08 element 协议设计

不适合并行的是：

- TP-01 与任意直接改 paste 入口的任务
- TP-03 schema 未冻结前的大规模 renderer 实现

### 8.4 解耦化实施原则

这次改造必须以“解耦、可替换、可灰度、可测试”为最高工程原则。

建议明确以下解耦约束：

1. 输入层不得直接创建画布元素  
   所有入口只允许产出 `Input Descriptor`，不允许在入口层继续分支创建 `text/image/fileCard`。

2. 解析层不得直接依赖画布运行时  
   parser 只能依赖输入描述对象、工具函数和 schema，不得直接读取 engine state 或 DOM 浮层状态。

3. canonical JSON 不得掺入视图状态  
   schema 只承载内容结构与来源元信息，不承载坐标、选区、缩放、菜单状态。

4. renderer 不得反向承担来源识别  
   renderer 只根据 canonical 节点工作，不能通过“原始来源类型”再做新一轮临时判断。

5. 元素层不得再承担解析职责  
   新元素协议只承接结构化结果，不再负责理解 HTML / Markdown / Office 原始格式。

6. 新旧链路必须通过网关和 fallback 协议连接  
   不允许直接在任何新模块里硬编码“失败后手动调用旧逻辑”的散乱回退。

7. 所有新增模块必须可单测  
   parser、schema、renderer、element adapter 都必须能脱离整张画布独立测试。

### 8.5 解耦化目录建议

建议尽量按职责拆目录，而不是按来源把逻辑散落到现有大文件中。

建议目录结构：

```text
public/src/engines/canvas2d-core/import/
  gateway/
  parsers/
    html/
    markdown/
    code/
    math/
    office/
    web/
  schema/
  renderers/
  diagnostics/
  fallbacks/
  adapters/
```

说明：

- `gateway/` 只收口入口，不做结构理解
- `parsers/` 只做来源解析
- `schema/` 只定义 canonical 协议
- `renderers/` 只做结构节点到元素的映射
- `diagnostics/` 只负责评分、日志、降级理由
- `fallbacks/` 只统一管理回退策略
- `adapters/` 只做新旧链路、schema 与 element 间的桥接

- 右键粘贴与 Ctrl + V 对比
- 超深层级列表
- 合并单元格表格
- 超长公式
- 超大代码块

### 验收标准

- 旧版基础文本复制粘贴不回归
- 新协议版本兼容测试通过
- 性能达到约定阈值

## TP-12 画布宿主改造与主链接管准备包

### 背景

到 `TP-11` 为止，新系统已经具备“统一输入、结构解析、协议表达、元素桥接、复制协议、灰度与观测”的能力。

但这仍然不等于“可以安全接管旧 paste / drag / clipboard 主链”。

真正的风险点已经从 parser 转移到了画布宿主侧：

- render plan 如何真正落板
- 新元素如何进入命中、搜索、导出、历史、存储
- 新元素如何在现有画布中稳定交互

因此在正式接管旧主链前，必须新增一个独立的宿主改造包。

### 目的

- 建立新系统接入现有画布宿主的统一承接层
- 补齐新元素的命中、编辑、搜索、导出、历史、存储配套能力
- 在 kill switch 保护下做小范围灰度接入实验

### 交付物

- Render Plan Commit Layer
- 新元素落板与布局回写规则
- 命中与交互宿主适配
- 编辑态宿主适配
- 搜索、导出、历史、存储宿主适配
- 小范围接主链灰度方案与回退验证

### 验收标准

- 至少一类主线内容已在真实入口下稳定走新系统
- 命中、编辑、搜索、导出、历史、存储具备最小闭环
- 出现问题时可通过 kill switch 一键退回旧主链
- 宿主接入不破坏现有旧主链默认行为

## 12. 风险重置

### 12.1 技术风险

- Office 富文本与 OMML 兼容复杂
- HTML 垃圾标签与网页结构波动大
- Markdown / GFM 与 HTML 混合输入时边界复杂
- 数学公式识别过激会误判普通文本
- 表格元素会显著提高命中、编辑、复制复杂度

### 12.1.1 高风险点

- 表格元素
- 任务列表与普通列表互转
- Markdown 与 HTML 混合输入
- 数学公式自动识别误判

### 12.1.2 中风险点

- 网页正文抽取
- 自定义解析器 / 渲染器接口
- Office 富文本中的局部样式与嵌套列表

### 12.2 工程风险

- 如果不先做兼容层隔离，会破坏现有主链
- 如果 parser / renderer 职责不清，会重新回到入口堆特判
- 如果元素层升级不到位，schema 的价值无法落地

### 12.2.1 建议的风险控制动作

- 每个任务包结束都要补样本
- 所有 parser 都必须有 fallback
- 所有 renderer 都必须有降级渲染
- 所有新增元素都必须定义“复制到旧链路”的降级行为

### 12.3 第三方依赖风险

- Markdown / GFM 解析库体积与兼容性
- KaTeX 或数学渲染库的字体、包体积、跨端差异
- 网页正文抽取准确率波动

### 12.4 产品风险

- 用户对旧粘贴行为已有习惯
- 新旧链路混用期，用户可能感知到不一致
- 某些极端内容在初期会先走降级逻辑

### 12.4.1 建议的产品层缓解策略

- 在开发版先暴露“新导入链路”开关
- 在失败降级时保留可见提示或日志
- 先让高价值来源走新链路，不要一次性全切

## 13. 最终结论

基于当前代码现状，这次改造不应该被理解为“增强复制粘贴”，而应该被理解为：

**把 FreeFlow 从“接收文本和 HTML 的画布”升级成“接收结构、理解结构、渲染结构的内容工作台”。**

当前项目并不是没有基础，反而已经具备：

- 可复用的入口
- 可复用的文本与图片元素
- 可复用的桌面剪贴板桥接
- 可复用的富文本编辑器能力

但它缺少的是中间层协议、解析器系统、渲染器系统和结构化元素体系。

因此，最合理的路线不是直接推翻旧版，而是：

1. 冻结旧链路
2. 建立 Import Gateway 与 canonical JSON
3. 优先打通 HTML / Markdown / 纯文本 / 代码 / LaTeX 主链
4. 同步升级元素层
5. 再逐步统一内部复制协议
6. 通过灰度与回退机制完成主链切换

这条路线是当前项目在稳定性、扩展性、长期能力三者之间最均衡的方案。

### 13.1 实施优先顺序一句话总结

最优顺序不是“先做所有 parser”，也不是“先做所有新元素”，而是：

**先统一入口和协议，再打通高价值来源，再补元素承接，再统一复制链路，最后灰度替换旧链路。**

## 14. 基于本次升级的全链路配套升级清单

本次结构化导入升级如果真正落地，影响的不会只有“画布元素”本身，而是整条内容链路。

因此，这次方案应被理解为一次“全链路升级”，而不是只改 parser 或只改 text 元素。

下面列出除主线协议外，必须同步评估与逐步升级的全链路模块。

### 14.1 画布元素体系升级

这是最核心的一环，也是最容易被低估的一环。

如果元素体系不升级，那么：

- Input Gateway 再统一
- Parser 再精细
- canonical JSON 再完整

最终也会在落地时被压回旧文本框模型。

因此，画布元素体系必须进行系统性升级。

重点包括：

- 文本元素协议增强
- 列表体系升级
- 任务列表支持
- 代码块元素
- 表格元素
- 数学公式元素
- 注脚 / 引用块的画布表达

建议原则：

- 不推翻全部旧元素
- 保留旧文本元素兼容能力
- 新增复杂结构专属元素
- 所有新元素都必须定义复制、命中、编辑、降级规则

### 14.2 输入入口层升级

除了画布元素，第一批必须同步升级的是所有内容进入画布的入口。

包括：

- Ctrl + V 粘贴
- 右键菜单粘贴
- 文件拖拽导入
- 文本 / HTML 拖拽导入
- 编辑器内粘贴
- 内部复制后再粘贴

升级目的：

- 不同入口最终统一进入 Import Gateway
- 不再出现“同样的内容从不同入口进来结果完全不同”

### 14.3 剪贴板协议层升级

本次升级必须同步重构剪贴板协议层。

包括：

- 内部复制 payload
- 系统剪贴板 `text/plain`
- 系统剪贴板 `text/html`
- 文件路径剪贴板
- 后续 canonical fragment 复制

升级目的：

- 让内部复制从“只复制元素”升级为“复制元素 + 复制结构片段”
- 让新元素复制到旧链路、外部应用时有稳定降级

### 14.4 导入网关与解析调度层升级

本次方案最直接依赖的基础设施包括：

- Import Gateway
- Source Detector
- Parser Registry
- Diagnostics / Scoring
- Fallback Strategy

如果这一层不升级，后续所有来源仍然会继续分散在入口层写特判。

### 14.5 富文本编辑器层升级

当前富文本编辑器层主要是轻量 HTML 编辑能力。

在结构化导入落地后，还需要进一步升级：

- 块级 / 行内级边界控制
- 任务列表编辑
- 代码块编辑体验
- 数学公式编辑体验
- 表格单元格编辑体验
- 与 canonical 协议的同步回写

升级目的：

- 不让结构化内容只能导入、不能稳定编辑

### 14.6 菜单与交互入口升级

右键菜单、悬浮工具条、更多菜单等都需要同步升级。

包括：

- 根据内容类型动态展示动作
- 支持“保真模式导入 / 结构模式导入”
- 对代码块 / 表格 / 公式 / 任务列表给出专属菜单项
- 提供结构转换与降级入口

升级目的：

- 让新结构内容在交互上真正可用，而不只是“被渲染出来”

### 14.7 渲染与布局系统升级

结构化导入会明显增加布局复杂度，因此渲染系统也必须升级。

包括：

- 混合内容流式布局
- 表格尺寸测量
- 代码块测量
- 数学公式尺寸测量
- 图文混排顺序控制
- 分批渲染与异步补全

升级目的：

- 避免结构正确但显示、测量、换行和重排不稳定

### 14.8 命中、选中、多选与拖拽系统升级

新元素落地后，命中与交互系统必须同步升级。

包括：

- 表格整体命中
- 单元格精准命中
- 代码块整体命中与文本区命中区分
- 公式整体命中与源码编辑区命中区分
- 任务列表复选框点击区与文本区区分
- 新元素在多选、框选、吸附、对齐线中的行为定义

升级目的：

- 保证新元素不仅“能显示”，而且“能像原生元素一样交互”

### 14.9 桌面端桥接能力升级

桌面端当前已有部分剪贴板桥接能力，但后续仍需加强。

包括：

- 系统剪贴板 HTML / 图片 / 文件统一桥接
- 文件路径与本地资源映射
- Office 来源识别辅助
- 数学公式来源识别辅助

升级目的：

- 让桌面端成为结构化导入的强能力环境，而不是被系统剪贴板能力卡住

### 14.10 数据持久化与兼容迁移升级

引入新元素和新协议后，持久化层必须同步升级。

包括：

- board JSON 如何存新元素
- schema version 版本迁移
- 老画布兼容读取
- 新旧内容混存策略
- 元素升级过程中的兼容字段处理

升级目的：

- 不让新结构内容在保存、读取、升级过程中失效

### 14.11 导出层升级

虽然本阶段主线不是导出，但导入结构升级后，导出迟早要跟上。

包括：

- 结构化内容导出 HTML
- 导出 Markdown
- 导出纯文本
- 新元素导出降级规则
- 后续导出 PDF / Word 的适配基础

升级目的：

- 不让 canonical 协议只服务“导入”，而无法服务“输出”

### 14.12 测试体系升级

本次升级必须同步建设测试体系。

包括：

- 样本库
- parser 测试
- renderer 测试
- schema 兼容测试
- 回归测试
- 真实粘贴 / 拖拽测试
- 性能压测

升级目的：

- 避免系统升级后只能靠手工试错

### 14.13 性能与异步任务系统升级

结构化导入能力增强后，性能问题会比现在更早暴露。

包括：

- 大 HTML 导入性能
- 大 Office 内容解析性能
- 大表格渲染性能
- 超长代码块渲染性能
- 超长公式渲染性能
- 分批处理、取消、加载状态反馈

升级目的：

- 让结构化能力在真实大内容下仍可用

### 14.14 设置、灰度与诊断体系升级

为了安全上线，设置和诊断系统要同步加强。

包括：

- 新导入链路开关
- 保真 / 结构模式开关
- 来源级功能开关
- parser 命中与降级日志
- 新旧链路差异日志
- 调试面板或开发日志输出

升级目的：

- 提高灰度切换、排障、回退效率

### 14.15 文档与扩展接口升级

除了代码，还要同步升级开发文档与扩展能力。

包括：

- canonical schema 文档
- parser 接口文档
- renderer 接口文档
- 样本规范
- 新元素协议文档
- 扩展接口版本说明

升级目的：

- 防止后续继续因为缺文档而回到“靠猜和复制逻辑开发”的状态

### 14.16 AI 预留层升级

虽然本阶段不纳入 AI 后处理，但结构化导入完成后，AI 层会成为自然延伸。

建议现在就预留：

- 节点级元数据
- 结构语义位
- 来源与诊断信息位
- 可供 AI 消费的块级标签

升级目的：

- 为下一阶段 AI 自动整理、拆节点、提取任务、摘要与转卡片打基础

### 14.17 全链路配套升级的总结论

基于这次结构化导入升级，真正需要同步优化的不是单一模块，而是：

- 画布元素体系
- 输入入口层
- 剪贴板协议层
- 导入网关与解析调度层
- 富文本编辑器层
- 菜单与交互入口
- 渲染与布局系统
- 命中、选中、多选与拖拽系统
- 桌面端桥接能力
- 数据持久化与兼容迁移
- 导出层
- 测试体系
- 性能与异步任务系统
- 设置、灰度与诊断体系
- 文档与扩展接口
- AI 预留层

因此，这次不应被理解为“只升级画布元素”，而应被理解为：

**一次从内容进入、结构理解、协议表达、元素落地到复制导出的全链路升级。**

### 14.18 接管旧主链前的画布宿主改造清单

在当前阶段，新系统已经具备：

- 统一输入协议
- parser / fallback / diagnostics 主链
- canonical JSON
- renderer pipeline
- 代码块 / 表格 / 数学公式等新元素协议
- 图片 / 文件 / 旧原生元素桥接
- copy / downgrade / 外部兼容输出
- rollout / kill switch / import log

但这仍然不等于“可以直接全面替换旧 paste / drag / clipboard 主链”。

真正接管旧主链之前，必须先补齐“画布宿主承接层”。否则问题会集中爆发在命中、编辑、复制、搜索、导出和历史系统，而不是 parser 本身。

建议把接管前的画布宿主改造清单明确为以下几类。

#### 14.18.1 Render Plan Commit Layer

必须新增一层明确的宿主承接逻辑：

- `canonical render plan -> board item commit`
- `legacy bridge plan -> old native item commit`
- `fallback output -> legacy import path commit`

目的：

- 不让 renderer 直接操作 engine store
- 不让旧主链入口继续承担新结构内容的最终落板职责

建议要求：

- commit layer 只负责“把已经确定的 render plan 落成 board items”
- 不再重新解析来源
- 不再临时追加内容结构判断

#### 14.18.2 新元素渲染承接与布局回写

需要明确这些内容真正如何进入现有画布：

- `text`
- `codeBlock`
- `table`
- `math`
- `image`
- `fileCard`
- 旧 native passthrough items

必须补的内容包括：

- 元素创建后的默认尺寸策略
- 初始布局位置策略
- 标题 / 段落 / 列表与普通文本的收敛策略
- 表格 / 代码块 / 公式的最小尺寸与自适应策略

目的：

- 防止新系统“能产出元素”，但落板后的布局体验不稳定

#### 14.18.3 命中与选中系统宿主升级

正式接主链前，至少要补这些命中模型：

- 表格整体命中
- 单元格精确命中
- 代码块整体命中与内容区命中区分
- 数学公式整体命中与源码编辑区命中区分
- task list checkbox 命中与文本区命中区分

目的：

- 保证新内容不是“只能显示”，而是“能被正确点中、框选、拖拽、编辑”

这部分如果不补，接主链后最先暴露的会是：

- 选不中
- 点中范围错位
- 框选不稳定
- 拖拽行为异常

#### 14.18.4 编辑态宿主升级

正式接入前，需要重新设计这些新元素的编辑语义：

- 代码块是否进入源码态
- 表格是否进入单元格编辑态
- 数学公式是否区分源码态与渲染态
- task list 是否支持勾选与文本编辑分离

目的：

- 避免新元素进入画布后只能作为“静态承接物”

建议原则：

- 文本、列表继续复用轻量编辑模型
- 代码块、表格、公式采用专属编辑态
- 所有编辑动作都要能回写到 `structuredImport` / canonical fragment

#### 14.18.5 搜索与索引宿主升级

正式接入前，需要把以下新元素纳入搜索索引：

- `codeBlock`
- `table`
- `math`
- `taskList`
- `image` 的标题/alt
- `fileCard` 的文件名/路径

目的：

- 保证导入进来的新结构内容不会变成“画布里存在但搜索看不见”

#### 14.18.6 导出与截图宿主升级

正式接入前，导出系统至少要知道如何处理：

- 代码块
- 表格
- 数学公式
- task list
- 新桥接图片与文件卡片

目的：

- 保证新元素在导出 PNG / PDF / board snapshot 时不丢失或退化异常

#### 14.18.7 历史记录与事务边界升级

正式接入前，历史栈至少要定义这些操作的事务边界：

- task list checked 切换
- 代码块源码修改
- 表格单元格编辑
- 公式源码修改
- 结构化导入整批落板

目的：

- 防止撤销 / 重做时出现“回退一半”或“回退粒度过细”的问题

建议原则：

- 单个结构块编辑尽量合并为一个事务
- 导入落板默认按一次导入事务收口

#### 14.18.8 存储与读取宿主升级

正式接入前，board 持久化层要确认：

- 新元素字段是否能稳定落盘
- `structuredImport` 与 `legacyCompatibility` 元数据是否保留
- 老 board 打开时是否兼容
- 新旧元素混存时是否有迁移歧义

目的：

- 避免主链接入后“当前能显示，但重新打开丢字段”

#### 14.18.9 复制、粘贴、拖拽回流宿主升级

正式接入前，要补宿主级回流规则：

- 新元素内部复制优先走 canonical fragment
- 外部复制优先走 external compatibility output
- 新元素拖拽回板时是否走 native passthrough 还是重建 structured fragment
- 右键粘贴与 `Ctrl + V` 是否统一走 switchboard + kill switch

目的：

- 不让新系统只在“第一次导入”有效，之后又退回旧分叉行为

#### 14.18.10 接管旧主链前的最小门槛

在决定让新系统正式接入旧主链之前，建议至少满足以下门槛：

1. `WP-39 ~ WP-42` 全部完成，并有稳定回归结果  
2. 至少一类主线内容已完成宿主承接闭环  
建议顺序：
- 先 `plain text / markdown text`
- 再 `image / fileCard`
- 最后 `codeBlock / table / math`
3. 搜索、导出、历史三条宿主配套能力至少具备最小支持  
4. `WP-36 / WP-37 / WP-38` 已接入实际灰度观测链  
5. 出现问题时可通过 kill switch 一键退回旧链路

#### 14.18.11 对当前阶段的判断

基于目前进度，可以明确判断为：

- 新系统已经具备“准备接主链”的条件
- 但还不具备“无风险全面替换旧主链”的条件

因此更合理的策略是：

- 先继续补宿主改造清单中的关键能力
- 再做小范围灰度接入
- 最后才做旧主链替换

换句话说，当前阶段最需要补的不是再发明新 parser，而是把“新内容进入现有画布后的宿主承接逻辑”设计和实现完整。

## 15. 后续升级方向

本节用于记录当前主线完成之后，FreeFlow 在结构化导入体系上值得继续推进的升级方向。

这些内容不建议混入当前第一阶段主线，但应尽早在架构上预留。

### 15.1 二期升级方向

二期建议优先围绕“来源覆盖扩展”和“结构稳定性增强”展开。

建议优先项：

- Office 数学公式深度支持
  - OMML 结构解析
  - MathML / LaTeX / OMML 多来源公式兼容
- Office 富文本增强
  - 更稳定的标题层级识别
  - 更稳定的段落间距与列表层级保留
  - 更复杂表格结构支持
- 网页正文增强
  - 更强的正文抽取
  - 页面噪声清理策略细化
  - 图文混排结构恢复增强
- Markdown 扩展增强
  - definition list
  - callout / admonition
  - 自定义块扩展
- 结构化元素增强
  - 列表从增强文本块进一步演进为独立元素
  - 更完整的注脚元素与关联交互
  - 更完整的引用块编辑能力

### 15.2 三期升级方向

三期建议围绕“统一内容协议平台化”和“跨链路一致性”展开。

建议优先项：

- 内外复制协议彻底统一
  - 内部元素复制
  - 外部导入
  - 编辑器内复制
  - 右键粘贴
  - 拖拽导入
  - 尽量共享同一 canonical 主链
- 反向导出协议
  - canonical JSON -> HTML
  - canonical JSON -> Markdown
  - canonical JSON -> 结构化 JSON 导出
- 结构化导出验真
  - 复制出去的 HTML / Markdown 尽量保持语义完整
  - 新元素复制到外部应用时不只“能贴出文本”，而是尽量保留结构
- 插件化扩展能力
  - 自定义 parser 发布机制
  - 自定义 renderer 发布机制
  - 自定义节点类型的注册协议

### 15.3 高阶升级方向

这部分不作为近期实施内容，但值得在协议设计时提前预留。

建议预留方向：

- 文档级语义块
  - 定义
  - 结论
  - 步骤
  - 警告
  - 任务
  - 参考资料
- 图文混排布局增强
  - 更复杂的块级编排
  - 关联说明块
  - 图表与正文联动
- 富媒体结构支持
  - 音频
  - 视频
  - 嵌入网页
  - 可交互附件
- 多端一致性
  - Web
  - 桌面端
  - 未来可能的移动端

### 15.4 与 AI 结合的后续方向

当前文档主线明确不纳入 AI 后处理，但从长期看，结构化导入架构完成后，AI 会成为下一阶段的自然延伸层。

建议后续单独立项，不要混入当前主线。

可能的方向包括：

- 自动把长文拆成节点结构
- 自动提炼标题与结论
- 自动识别任务项
- 自动识别表格含义并转数据卡片
- 自动识别代码块语言与用途
- 自动整理图文顺序与块级布局

### 15.5 协议预留建议

为了支持后续升级，当前第一阶段就建议预留这些设计余量：

- canonical JSON 节点元数据区
- 自定义节点类型命名空间
- 节点级扩展字段
- parser / renderer 接口版本号
- 诊断信息扩展位

### 15.6 后续升级的总原则

后续不管扩什么，都建议坚持这几条原则：

- 不绕过 Import Gateway
- 不绕过 canonical JSON
- 不让外部格式直接成为画布最终协议
- 不让单个文本元素继续吞掉所有复杂结构
- 新来源优先新增 parser，不优先改核心 renderer
- 新结构优先新增 renderer / element，不优先改旧文本特判
