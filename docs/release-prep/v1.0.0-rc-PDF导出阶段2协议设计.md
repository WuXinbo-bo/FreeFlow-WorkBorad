# v1.0.0-rc PDF 导出阶段2协议设计

## 阶段目标

本阶段只定义协议，不进入功能实现。

目标是把以下内容定死：

- PDF 导出配置对象
- PDF 导出返回结果对象
- 异常策略与错误码
- 桌面端 / Web 端保存策略
- 超大画布降级规则

本阶段所有协议都围绕以下约束展开：

- 不污染现有画布数据结构
- 不改动画布元素存储格式
- 不干扰自动保存、教程、最近画布、复制粘贴主链路
- 不把 PDF 配置持久化进画布 JSON

---

## 一、协议设计原则

### 1. 与现有图片导出链路保持一致

现有导出链路已经有以下模式：

- 输入一组元素或视口
- 渲染成离屏 canvas
- 转成 dataUrl
- 走桌面端 `pick save path + writeFile` 或 web 端 `Blob + download`
- 用 `setStatus()` 反馈导出结果

PDF 导出应延续这套交互语义，但新增独立协议，而不是硬塞进现有图片导出函数。

### 2. 协议只服务运行时，不进入持久化

PDF 导出配置只属于“本次导出动作”：

- 不写进 `state.board`
- 不写进 board JSON
- 不写进 autosave
- 不写进最近画布

### 3. 第一版只支持单页 PDF

阶段2明确锁定：

- 只做单页 PDF
- 只做整张画布导出
- 不做分页
- 不做矢量 PDF
- 不做导出状态恢复

## 二、建议的配置对象

建议使用独立配置对象，例如：

```js
/**
 * @typedef {Object} PdfExportOptions
 * @property {"board"} scope
 * @property {1|2|3} scale
 * @property {"white"|"transparent"} background
 * @property {boolean} includeGrid
 * @property {"fit-single-page"} pageMode
 * @property {string} fileName
 * @property {"auto"|"portrait"|"landscape"} orientation
 * @property {"png"} imageFormat
 * @property {boolean} preferDownload
 */
```

### 字段定义

#### 1. `scope`

- 类型：`"board"`
- 默认值：`"board"`
- 第一版只允许一个值

设计理由：

- 当前阶段只做整张画布导出
- 以后扩展当前视口、选区、多页时再扩枚举

#### 2. `scale`

- 类型：`1 | 2 | 3`
- 默认值：`2`

设计理由：

- `1`：标准清晰度
- `2`：默认，兼顾清晰度和性能
- `3`：高清

不建议阶段2就开放任意浮点数：

- 现有导出与渲染逻辑更适合固定档位
- 更容易做超大画布降级策略

#### 3. `background`

- 类型：`"white" | "transparent"`
- 默认值：`"white"`

设计理由：

- PDF 第一版更适合白底，兼容打印与阅读
- 透明背景虽然技术上可做，但落到不同 PDF 阅读器体验不稳定
- 白底也和现有 `forceWhiteBackground` 的图片导出习惯一致

#### 4. `includeGrid`

- 类型：`boolean`
- 默认值：`false`

设计理由：

- 网格属于画布辅助信息，不适合默认进入正式导出件
- 但保留协议位，后续不必再破协议

#### 5. `pageMode`

- 类型：`"fit-single-page"`
- 默认值：`"fit-single-page"`

设计理由：

- 阶段2先把单页适配语义锁住
- 以后若做多页，可扩：
  - `paginate`
  - `fit-width`
  - `fit-height`

#### 6. `fileName`

- 类型：`string`
- 默认值：当前画布名，fallback 为 `freeflow-board`

设计理由：

- 现有图片与文本导出都有 `defaultName`
- PDF 导出也应保留统一的命名入口

建议命名规则：

- 去掉非法字符
- 自动补 `.pdf`

#### 7. `orientation`

- 类型：`"auto" | "portrait" | "landscape"`
- 默认值：`"auto"`

设计理由：

- 单页 PDF 需要确定页面朝向
- `auto` 最适合第一版，按画布宽高比自动决定

#### 8. `imageFormat`

- 类型：`"png"`
- 默认值：`"png"`

设计理由：

- 第一版建议锁成 `png`
- 原因是当前画布包含文本、线条、富文本、图片卡片、文件卡，PNG 的细节保真更稳
- 后续若要压缩体积，再扩 `jpeg`

#### 9. `preferDownload`

- 类型：`boolean`
- 默认值：`false`

设计理由：

- 桌面端优先原生保存
- Web 端优先下载
- 这个字段只是保留后续空间，第一版可以内部固定策略，不一定对 UI 暴露

## 三、建议的返回结果对象

建议使用统一返回结构：

```js
/**
 * @typedef {Object} PdfExportResult
 * @property {boolean} ok
 * @property {boolean} canceled
 * @property {string} code
 * @property {string} message
 * @property {string} fileName
 * @property {string} filePath
 * @property {number} bytes
 * @property {number} pageWidth
 * @property {number} pageHeight
 * @property {number} scaleApplied
 */
```

### 字段说明

#### 1. `ok`

- 导出是否成功

#### 2. `canceled`

- 是否为用户主动取消保存
- 与失败区分开

#### 3. `code`

- 统一错误码 / 成功码
- 不建议只靠 message 判定

#### 4. `message`

- 给 UI 的直接提示文案

#### 5. `fileName`

- 最终导出文件名

#### 6. `filePath`

- 桌面端返回真实保存路径
- Web 端可以为空字符串

#### 7. `bytes`

- 导出 PDF 字节大小
- 方便后续调试、日志、体积限制判断

#### 8. `pageWidth` / `pageHeight`

- 最终 PDF 页面逻辑尺寸

#### 9. `scaleApplied`

- 实际应用的导出倍率
- 超大画布时可能从 `3` 自动降到 `2` 或 `1`

## 四、建议的错误码

建议错误码保持简洁、面向导出链路：

### 成功类

- `PDF_EXPORT_OK`
- `PDF_EXPORT_OK_DOWNGRADED`

### 取消类

- `PDF_EXPORT_CANCELED`

### 数据与内容类

- `PDF_EXPORT_EMPTY_BOARD`
- `PDF_EXPORT_INVALID_OPTIONS`
- `PDF_EXPORT_INVALID_FILENAME`

### 渲染类

- `PDF_EXPORT_HYDRATE_FAILED`
- `PDF_EXPORT_PRELOAD_FAILED`
- `PDF_EXPORT_RENDER_FAILED`
- `PDF_EXPORT_CANVAS_EMPTY`
- `PDF_EXPORT_CANVAS_TAINTED`

### 性能与限制类

- `PDF_EXPORT_TOO_LARGE`
- `PDF_EXPORT_SCALE_DOWNGRADED`
- `PDF_EXPORT_MEMORY_RISK`

### 保存类

- `PDF_EXPORT_SAVE_UNSUPPORTED`
- `PDF_EXPORT_PICK_PATH_FAILED`
- `PDF_EXPORT_WRITE_FAILED`
- `PDF_EXPORT_DOWNLOAD_FAILED`

### 未知类

- `PDF_EXPORT_UNKNOWN_ERROR`

- `导出失败：当前环境不支持 PDF 导出`

## 六、桌面端 / Web 端保存策略

### 1. 桌面端策略

建议复用现有保存思路：

- 优先 `desktopShell.pick...SavePath`
- 再用 `desktopShell.writeFile`

当前已有：

- 图片：`pickImageSavePath`
- 文本：`pickTextSavePath`

阶段2建议新增对应 PDF 能力时采用以下策略之一：

#### 方案 A：新增原生 PDF 保存选择器

- `desktopShell.pickPdfSavePath({ defaultName })`

优点：

- 语义最清晰

#### 方案 B：通用文件保存选择器

- `desktopShell.pickSavePath({ defaultName, filters })`

优点：

- 后续更通用

阶段2结论：

- 协议上应预留“桌面端原生选路径 + writeFile”的模式
- 不建议复用 `pickImageSavePath` 伪装保存 PDF

### 2. Web 端策略

建议沿用现有图片导出的下载模式：

- `Blob`
- `URL.createObjectURL`
- `a.download`

这是当前项目最一致、最轻的方案。

## 七、超大画布降级规则

阶段2先把策略定死，阶段3/4再实现。

### 建议规则

#### 1. 正常范围

- 使用用户请求的 `scale`

#### 2. 中度超限

- 自动将 `3 -> 2`
- 自动将 `2 -> 1`
- 返回：
  - `ok: true`
  - `code: PDF_EXPORT_OK_DOWNGRADED`
  - `scaleApplied` 为实际倍率

#### 3. 严重超限

- 直接失败
- 返回：
  - `ok: false`
  - `code: PDF_EXPORT_TOO_LARGE`

### 降级触发维度建议

- 总像素数
- 单边最大尺寸
- 估算字节量

### 第一版建议原则

- 先做“自动降级”
- 不做“后台分块拼接”
- 不做“自动分页 PDF”

## 十、阶段2结论

阶段2协议建议最终收敛为：

1. 配置对象只描述本次 PDF 导出动作，不进入持久化。
2. 返回对象必须包含 `ok/canceled/code/message` 四个基础字段。
3. UI 提示统一走 `setStatus()`，不混用画布保存 toast。
4. 桌面端保存必须走原生选路径 + `writeFile` 语义。
5. Web 端保存沿用 `Blob + download`。
6. 超大画布第一版只做自动降级和失败保护，不做分页。

## 附录 A：文本尺寸任务包阶段 1 执行结果

本附录用于冻结“文本尺寸统一测量”改造前的现状基线，作为后续文本框收敛与 PDF 边界改造的依据。

### A.1 文本尺寸链路说明

#### 1. 新建文本

- 新建文本元素走 [createTextElement()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:99)
- 初始宽高来自 [measureTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:52)
- 该路径是启发式估算：
  - 以最长行字符数估宽
  - 以行数和字号估高
- 这意味着空白画布上第一次创建文本时，`auto-width` 文本框并不是按真实富文本排版结果生成

#### 2. 编辑中实时尺寸

- 编辑态文本测量走 [getEditingTextBoxMetrics()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:162)
- 测量优先读取真实编辑器 DOM：
  - `scrollWidth`
  - `scrollHeight`
  - `getBoundingClientRect()`
- 如果 DOM 不可用，会回退到 [getTextMinSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:64)
- 编辑中回写逻辑在 [syncEditingTextItemSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:2569)

#### 3. 提交编辑后尺寸

- 文本提交编辑时，会先执行 [syncEditingTextItemSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:2569)，再完成提交
- 文本提交入口位于 [createCanvas2DEngine.js](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:2792)
- 当前实际行为：
  - 编辑态用 DOM 尺寸驱动 `item.width / item.height`
  - 提交后未编辑态直接沿用这个尺寸

#### 4. 未编辑态渲染使用尺寸

- 未编辑态绘制入口在 [drawTextElement()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js:467)
- 实际渲染直接使用元素现有的：
  - `item.width`
  - `item.height`
- 未编辑态本身不会在每次渲染前重新按真实内容做精确收敛

#### 5. 字号与文本预设变化后的尺寸

- 富文本工具栏字号变化发生在编辑态，仍通过编辑器内容变化触发 [syncEditingTextItemSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:2569)
- 非编辑态文本预设切换走 [applyTextPresetToSelection()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:6497)
- 预设切换后会调用 [getTextMinSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:64) 重算尺寸
- 这意味着：
  - 文本预设切换已具备“尺寸重算”链路
  - 但 `auto-width` 仍主要依赖启发式估算

#### 6. 系统粘贴文本后的尺寸

- 画布级系统粘贴入口是 [pasteFromSystemClipboard()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:7007)
- 纯文本粘贴会进入 [insertTextAt()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:4400)
- 再走 [dragBroker.createElementsFromText()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/brokers/dragBroker.js:50)
- 最终仍使用 [createTextElement()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:99) 的启发式初始尺寸

#### 7. 编辑器内粘贴富文本后的尺寸

- 富文本编辑器内部粘贴入口在 [richTextAdapter.js](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/editors/richTextAdapter.js:484)
- 粘贴 HTML 会插入规范化后的富文本内容
- 粘贴后的尺寸更新仍依赖编辑态的 [syncEditingTextItemSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:2569)
- 所以“系统粘贴到画布”和“在编辑器里粘贴富文本”是两条不同链路：
  - 前者先创建文本元素，再用启发式初始尺寸
  - 后者发生在编辑态，尺寸更接近真实 DOM 测量

#### 8. PDF 导出使用边界

- PDF 导出入口在 [exportBoardAsPdf()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:3628)
- 离屏渲染入口在 [renderBoardToCanvas()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:3602)
- 导出边界通过 [getExportBounds()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/export/renderBoardToCanvas.js:21) 聚合元素外框
- 对文本元素来说，边界本质就是现有的 `item.width / item.height`
- 因此文本框尺寸偏大，会直接放大 PDF 导出边界

### A.2 当前模式分叉

#### 1. `auto-width`

- 主入口：
  - [getTextMinSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:64)
  - [measureTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:52)
- 当前策略：
  - 宽高主路径依赖字符数启发式估算
  - 非编辑态是当前误差最大的模式

#### 2. `wrap`

- 主入口：
  - [getTextMinSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:64)
  - [measureRichTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js:349)
- 当前策略：
  - 宽度使用 `widthHint`
  - 高度按富文本 runs 的 canvas 布局结果估算
- 当前问题不是完全没有真实测量，而是尚未和编辑态 DOM 测量统一

#### 3. `flowNode`

- 主入口：
  - [getFlowNodeMinSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/flow.js:11)
- 当前策略：
  - 采用固定最小尺寸
  - 基于 `measureRichTextBox()` 计算内容区最小宽高
- `flowNode` 已经不走 `measureTextBox()`，但和文本元素仍是独立测量链

### A.3 当前问题结论

当前项目的文本尺寸来源是分裂的：

- 新建文本和系统粘贴文本：启发式初始尺寸
- 编辑态：真实 DOM 测量优先
- 非编辑态 `auto-width`：启发式估算
- 非编辑态 `wrap` / `flowNode`：canvas runs 测量
- PDF 导出：直接消费现有元素外框

因此当前最核心的问题不是“所有文本都没有测量能力”，而是：

**同一个文本元素在不同阶段使用了不同的尺寸来源，尤其 `auto-width` 未编辑态与编辑态不一致。**

## 附录 B：文本尺寸任务包阶段 2 协议草案

本附录定义文本尺寸统一测量接口，只锁协议，不进入功能实现。

### B.1 设计目标

统一以下场景的文本尺寸来源：

- 新建文本
- 编辑中实时尺寸兜底
- 编辑完成后尺寸回写
- 系统粘贴文本后尺寸
- 预设与字号变化后的尺寸
- 导出前文本边界校正

### B.2 当前项目已具备的测量能力

#### 1. DOM 编辑器测量

- 入口：[getEditingTextBoxMetrics()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:162)
- 特征：
  - 读取编辑器 DOM 尺寸
  - 与真实输入状态最接近
  - 当前只服务编辑态

#### 2. canvas runs 富文本测量

- 入口：[measureRichTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js:349)
- 特征：
  - 解析 HTML 为 runs
  - 用 canvas `measureText` 做布局
  - 适合 `wrap` 和 `flowNode`

#### 3. 启发式文本测量

- 入口：[measureTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:52)
- 特征：
  - 速度快
  - 只感知字符数和字号
  - 适合作为最后兜底，不适合作为主链路

### B.3 统一测量接口

建议接口：

```js
measureTextElementLayout({
  html,
  plainText,
  fontSize,
  resizeMode,
  widthHint,
  lineHeightRatio,
  fontWeight,
  boldWeight,
})
```

### B.4 建议输入字段

#### 必需输入

- `html`
- `plainText`
- `fontSize`
- `resizeMode`

#### 条件输入

- `widthHint`
  - `wrap` 和 `flowNode` 模式需要
- `lineHeightRatio`
  - 用于保持与当前渲染链一致
- `fontWeight`
- `boldWeight`
  - 用于保持与富文本 runs 测量和显示样式一致

### B.5 建议返回结构

建议统一返回：

```js
{
  contentWidth,
  contentHeight,
  frameWidth,
  frameHeight,
  measuredWith,
  reason,
}
```

字段说明：

- `contentWidth`
  - 纯内容自然宽度
- `contentHeight`
  - 纯内容自然高度
- `frameWidth`
  - 应用于元素外框的宽度
- `frameHeight`
  - 应用于元素外框的高度
- `measuredWith`
  - `"dom" | "canvas" | "fallback"`
- `reason`
  - 说明此次结果属于 `auto-width`、`wrap`、空文本、降级等哪种情况

### B.6 模式规则

#### 1. `auto-width`

- 宽度按内容自然宽度收敛
- 高度按自然内容高度收敛
- 不再以 `measureTextBox()` 作为主路径

#### 2. `wrap`

- 宽度以 `widthHint` 为准
- 高度按真实换行结果收敛
- 不主动改写用户当前有效宽度策略

#### 3. `flowNode`

- 保留节点最小宽高约束
- 内容测量接口可复用统一测量层
- 但外框仍允许叠加节点自身 padding 和最小尺寸规则

### B.7 回退顺序

建议严格按以下顺序回退：

1. DOM 测量
   - 优先用于有 `document` 且可安全创建测量 host 的运行环境
2. canvas/runs 测量
   - 复用 `measureRichTextBox()`
3. 启发式 fallback
   - 复用 `measureTextBox()`

补充原则：

- `measureTextBox()` 只作为失败兜底，不再作为 `auto-width` 正常主路径
- DOM 和 canvas 结果不要求完全相同，但必须保持可解释的一致性

### B.8 副作用约束

统一测量层必须满足：

- 不写回 `state.board`
- 不触发 `store.emit()`
- 不触发 autosave
- 不修改当前选中态
- 不修改编辑器内容
- 不依赖当前画布 view
- 不修改导出链路输入数据

### B.9 阶段 2 结论

阶段 2 的协议应最终锁定为：

1. 文本测量必须抽成独立能力，而不是分散在编辑器、渲染器、导出器里各算一遍。
2. `auto-width`、`wrap`、`flowNode` 共享同一个测量接口，但保留各自外框规则。
3. DOM 测量优先，canvas runs 次之，启发式最后兜底。
4. 返回值必须同时区分“内容尺寸”和“外框尺寸”，为后续导出边界复用留接口。
5. 统一测量层必须是无副作用的纯测量模块。

## 附录 C：文本尺寸任务包阶段 3 设计草案

本附录定义“统一测量层”的模块设计，只描述结构、职责和接线边界，不进入功能实现。

### C.1 阶段目标

把文本尺寸测量从以下现有分散位置中抽离出来：

- 编辑态 DOM 测量
- `measureRichTextBox()` 的 canvas runs 测量
- `measureTextBox()` 的启发式估算

抽离后的统一测量层只负责一件事：

**根据输入文本内容与模式，返回稳定、无副作用的布局测量结果。**

### C.2 当前适合复用的底层能力

#### 1. 富文本 runs 解析与 canvas 布局

当前最适合复用的底层能力在 [rendererText.js](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js)：

- [parseRichTextRuns()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js:76)
- [fontForRun()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js:127)
- [layoutRichRuns()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js:206)
- [measureRichTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/rendererText.js:349)

这些能力说明：

- 当前项目已经具备“富文本 HTML -> runs -> 版面布局 -> 宽高结果”的基础能力
- 阶段 3 不需要从零重写富文本布局器
- 更合理的方向是把它从“渲染器附属能力”提升为“统一测量层的一部分”

#### 2. 文本启发式兜底能力

当前可直接复用的兜底能力在 [measureTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:52)。

阶段 3 的目标不是删除它，而是：

- 保留它作为最后兜底
- 不再让它承担 `auto-width` 的主路径

### C.3 建议模块拆分

建议新增目录：

```txt
public/src/engines/canvas2d-core/textLayout/
```

建议新增文件：

#### 1. `measureTextElementLayout.js`

职责：

- 统一接收文本测量输入
- 调度 DOM / canvas / fallback 三种测量方式
- 输出阶段 2 约定的统一结果对象

建议它成为后续所有文本尺寸计算的唯一入口。

#### 2. `createTextMeasureHost.js`

职责：

- 创建隐藏 measurement host
- 管理可复用的 DOM 测量节点
- 保证测量 host 与真实编辑器或显示样式尽量一致

该模块不负责业务逻辑，只负责 host 生命周期与基础样式。

#### 3. `measureTextWithDom.js`

职责：

- 接收统一协议输入
- 使用 measurement host 注入 HTML
- 读取自然内容尺寸与外框尺寸

它应当是新的第一优先级测量器。

#### 4. `measureTextWithCanvas.js`

职责：

- 基于 `measureRichTextBox()` 和现有 runs 逻辑做 canvas 测量
- 为 DOM 不可用或不可靠时提供通用 fallback

#### 5. `measureTextWithFallback.js`

职责：

- 封装 `measureTextBox()`
- 只在 DOM/canvas 都不可用时返回兜底结果

### C.4 建议依赖关系

建议依赖层次如下：

```txt
measureTextElementLayout
  -> measureTextWithDom
  -> measureTextWithCanvas
  -> measureTextWithFallback
```

限制：

- `createCanvas2DEngine.js` 不能直接依赖 DOM 测量细节
- `text.js` 不能继续直接拼装多套测量策略
- `rendererText.js` 继续保留渲染职责，但其内部部分布局能力应作为测量层依赖被复用

### C.5 DOM measurement host 约束

新的 DOM 测量 host 必须满足：

#### 1. 隐藏但可测量

- 不能 `display: none`
- 应采用离屏或不可见但仍参与布局的方案

#### 2. 样式尽量贴近真实显示

至少要对齐：

- `font-size`
- `font-weight`
- `line-height`
- 段落和列表的基本块级规则
- 编辑器和未编辑态都在意的文本内边距策略

#### 3. 不复用真实编辑器节点

原因：

- 当前真实编辑器带 selection/focus/input 副作用
- 阶段 3 的 measurement host 必须是“纯测量容器”

#### 4. 可复用，不按每次测量都创建完整 DOM 树

原因：

- 后续阶段 5、阶段 6 会频繁调用统一测量
- host 必须能被重用，否则会引入明显性能损耗

### C.6 统一测量层的输出规则

阶段 3 应先保证这些规则成立：

#### 1. 对 `auto-width`

- 返回真实内容自然宽度
- 返回真实内容自然高度
- 外框尺寸默认等于内容尺寸，后续再由调用方决定是否加最小约束

#### 2. 对 `wrap`

- 使用 `widthHint` 计算换行结果
- 返回内容高度
- 外框宽度沿用 `widthHint`

#### 3. 对 `flowNode`

- 测量层只负责内容尺寸
- 节点 padding、最小宽高仍由 flowNode 规则叠加

### C.7 阶段 3 与现有文件的接缝

本阶段不直接改功能逻辑，但后续要准备好这些接缝：

#### `text.js`

- 当前入口：
  - [getTextMinSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:64)
  - [createTextElement()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:99)
  - [normalizeTextElement()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:123)
- 后续会改为消费统一测量层

#### `flow.js`

- 当前入口：
  - [getFlowNodeMinSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/flow.js:13)
- 后续会复用统一测量层的内容尺寸结果

#### `createCanvas2DEngine.js`

- 当前入口：
  - [getEditingTextBoxMetrics()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/createCanvas2DEngine.js:162)
- 后续可逐步退化为“编辑态专用 DOM 读数器”或直接改为调用统一测量层

### C.8 阶段 3 产出物

阶段 3 完成后，应具备以下产物：

1. 一个统一测量入口模块
2. 一个可复用 DOM measurement host
3. 一个桥接 canvas/runs 的测量模块
4. 一个桥接启发式 fallback 的测量模块
5. 一份明确的依赖边界说明

### C.9 阶段 3 结论

阶段 3 的重点不是立刻改所有调用方，而是先把“测量能力”独立出来。

只有统一测量层先成立，阶段 4 才能安全接管文本尺寸主入口，而不会继续让 `text.js`、编辑器、导出器各自维护一套规则。

## 附录 D：文本尺寸任务包阶段 4 设计草案

本附录定义“文本尺寸主入口接管”方案，即在不改 JSON 结构的前提下，把文本元素宽高逐步切换到统一测量层。

### D.1 阶段目标

让以下三个关键入口不再主要依赖启发式估算：

- [getTextMinSize()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:64)
- [createTextElement()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:99)
- [normalizeTextElement()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:123)

### D.2 接管原则

#### 1. 不先改存储结构

- 仍然沿用 `item.width / item.height`
- 不新增持久化字段
- 不引入迁移字段

#### 2. 先替换尺寸来源，再考虑旧数据迁移

阶段 4 只做：

- 新建文本更准
- 标准化更准
- `getTextMinSize()` 更准

不在本阶段做：

- 画布打开时全量重测
- 历史文本自动回写

#### 3. `measureTextBox()` 继续保留，但降为最后兜底

这一步必须明确，否则阶段 4 改完仍然可能在主路径回落到旧问题。

### D.3 `getTextMinSize()` 的改造要求

当前问题点：

- `auto-width` 直接走 [measureTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:70)
- `wrap` 直接走 [measureRichTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:78)

阶段 4 改造后应变成：

#### 1. `auto-width`

- 主路径调用 `measureTextElementLayout(...)`
- 取其 `frameWidth / frameHeight`
- 仅在统一测量失败时回退到 `measureTextBox()`

#### 2. `wrap`

- 主路径也调用 `measureTextElementLayout(...)`
- 传入 `widthHint`
- 宽度仍以 `widthHint` 为准
- 高度取统一测量层返回结果

#### 3. 返回值语义保持兼容

为了降低改动范围，阶段 4 仍可保持：

```js
{ width, height }
```

但其内部来源应改为：

- 统一测量层结果优先
- 启发式结果仅作兜底

### D.4 `normalizeTextElement()` 的改造要求

当前问题点：

- `auto-width` 宽度会重新估算，但不是按真实排版
- 高度存在旧值优先保留问题，见 [text.js](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:150)

阶段 4 要求：

#### 1. 标准化阶段改为使用统一测量结果

- `auto-width`：
  - 宽度来自统一测量层
  - 高度来自统一测量层
- `wrap`：
  - 宽度优先保留已有有效宽度
  - 高度来自统一测量层

#### 2. 旧值保留策略收紧

标准化时不要再简单使用：

- `Number(element.height) || metrics.height`

而应按模式判断：

- `auto-width` 可以更积极使用新测量结果
- `wrap` 主要保留宽度，不机械保留旧高度

#### 3. 保证兼容旧画布

- 仍保留 `inferLegacyTextResizeMode(...)`
- 先不移除旧兼容逻辑
- 避免阶段 4 直接破坏历史数据识别

### D.5 `createTextElement()` 的改造要求

当前问题点：

- 初始尺寸完全来自 [measureTextBox()](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/text.js:102)

阶段 4 要求：

- 新建文本应优先通过统一测量层得到初始尺寸
- 如果内容为空，应保持安全的最小默认框
- 如果是从 HTML 或粘贴富文本创建，应优先按 HTML 测量结果生成尺寸

这样可以减少“文本刚创建时就已经偏大”的问题。

### D.6 `flowNode` 在阶段 4 的位置

阶段 4 的主攻对象是文本元素，但需要提前定义 flowNode 的处理边界：

- 不要求在阶段 4 重写 `getFlowNodeMinSize()`
- 但 flowNode 的内容测量能力应准备复用统一测量层
- flowNode 的最小宽高、padding、节点结构约束继续留在 [flow.js](/d:/FreeFlow-WorkBoard/public/src/engines/canvas2d-core/elements/flow.js:13)

也就是说：

**阶段 4 先让普通文本接管统一测量层，flowNode 在接口层对齐，在规则层暂不完全并轨。**

### D.7 阶段 4 需要修改的文件范围

阶段 4 预期主要影响：

#### 1. `elements/text.js`

重点接管：

- `getTextMinSize()`
- `createTextElement()`
- `normalizeTextElement()`

#### 2. `rendererText.js`

本阶段不应让它承担新的业务入口，但可能需要：

- 导出内部布局工具
- 暴露可复用的 runs 测量能力

#### 3. `elements/flow.js`

仅做接口层预留，不应在阶段 4 做过量改造。

### D.8 阶段 4 的验收标准

阶段 4 完成后，应至少满足：

1. 新建文本框尺寸更接近真实内容
2. `auto-width` 标准化结果不再主要依赖字符数估算
3. `wrap` 高度统一通过新测量层得到
4. 历史文本不会因为阶段 4 而大面积破坏布局
5. 不引入新的持久化字段

### D.9 阶段 4 结论

阶段 4 的本质不是“彻底修完所有文本问题”，而是：

**把文本尺寸主入口从旧的启发式中心，切换到新的统一测量中心。**

一旦这一步完成，后续阶段 5 才能顺利接管编辑完成、粘贴完成、字号变化和预设变化后的尺寸回写。
