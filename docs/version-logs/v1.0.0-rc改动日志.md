# FreeFlow v1.0.0-rc 改动日志

## 版本信息

- 目标版本：`v1.0.0-rc`
- 日志用途：记录 `v1.0.0-rc` 版本周期内的界面调整、Bug 修复、交互修正与结构性优化
- 维护方式：后续所有 `v1.0.0-rc` 改动统一追加到本文件

## 版本迭代总结

- 桌面工作台结构重构：
  - 完成左右主面板布局底座、自由拖拽缩放、左右换位、浮层解耦与全局 overlay 分层整理，整体桌面端结构比 `v0.8.0-beta` 更稳定。
- 桌面窗口 shape 主线收敛：
  - 持续重做透明窗口可点击区域收集、同步与调度机制，修复透明区失去穿透、浮层显示与点击区域不同步、教程宿主误占整窗等一系列根问题。
- 画布引擎与交互增强：
  - 重点补齐最近画布/教程画布启动链路、空白区右键菜单与拖动画布区分、多选复制粘贴排版保留、富文本编辑与工具条适配、画布内全局搜索、快捷键说明接入教程中心等核心体验问题。
- 教程系统重搭与内容整理：
  - 将教程中心、主界面教程、画布教程、AI 镜像教程重新梳理，并把快捷键速查改为独立说明文档入口，减少旧版教程链路反复叠加后的臃肿逻辑。
- 启动与兼容性修复：
  - 修复多处启动顺序错误、历史布局迁移导致的正式界面打不开、变量解构缺失引起的启动页卡死，以及旧用户升级后最近画布/教程画布判断异常等问题。
- AI 镜像与顶部控制区稳定性提升：
  - 调整映射控制、设置、更多菜单、全屏与嵌入相关控件的层级、定位和恢复链路，减少右侧界面拉伸、关闭和切换时的挤压、错位与失效。
- 版本定位提升：
  - 本轮改动已明显超出 `v0.8.1` 的局部修补范围，更接近一次面向正式版的结构收敛与体验补完，因此版本号正式提升为 `v1.0.0-rc`。

## 新增功能简述

- 自由布局主面板：
  - 左右主界面支持拖拽移动、独立缩放、上下位移和左右换位，桌面工作区布局更灵活。
- 全局浮层统一层：
  - 设置抽屉、更多菜单、映射控制、教程中心等浮层统一迁入全局 overlay 层，显示与交互更稳定。
- 教程中心重构：
  - 新增更清晰的教程中心分区，主界面教程、画布教程、AI 镜像教程分开管理。
- 快捷键速查入口：
  - 新增画布与界面快捷键速查文档入口，用户可以直接查看常用快捷键，不必进入引导流程。
- 画布内全局搜索：
  - 新增画布内容全局搜索与结果定位能力，便于在复杂画布中快速查找目标元素与文本信息。
- 画布基础操作增强：
  - 新增或补强空白区右键菜单、右键拖动画布、中键拖动、视角缩放等基础交互说明与行为一致性。
- 富文本编辑能力增强：
  - 富文本工具条布局更稳定，字号限制放开，编辑体验与大字号场景适配更完整。
- 启动画布逻辑优化：
  - 教程画布回归首次引导用途，最近画布恢复链路单独梳理，减少反复打开教程画布的问题。

## 本轮已记录改动

### 1. 画布悬浮 UI 自适应逻辑修正

- 调整对象：
  - 左上角信息面板
  - 右上角工具栏面板
  - 右下角缩放面板
- 原始问题：
  - 初版误将悬浮 UI 缩放逻辑绑定到“画布视角缩放”
  - 用户真实需求是：在窗口左右拉伸、全屏/非全屏切换时，保持悬浮 UI 尺寸一致性并避免重叠
- 修正结果：
  - 已取消“跟随画布视角缩放”的错误联动
  - 改为根据画布 UI 容器实际宽高变化动态计算悬浮面板缩放系数
  - 宽度充足时保持原尺寸
  - 宽度不足、存在挤压风险时再按阈值缩小，降低左上与右上区域重叠概率
  - 保留最小缩放限制，避免缩小后不可操作
- 影响范围：
  - 仅调整悬浮 UI 的响应式呈现逻辑
  - 不修改画布引擎内部绘制、视角、交互功能
- 相关文件：
  - `public/src/engines/canvas2d-core/ui/index.jsx`
  - `public/styles.css`
  - `public/assets/canvas2d-ui/current/canvas2d-ui.js`

### 2. 右上角设置 / 更多按钮独立浮层化

- 类型：Bug 修复 / 交互结构调整
- 原始问题：
  - 右侧上方的“设置”“更多”按钮原本挂在右侧对话面板头部结构中
  - 当左右界面拉伸、收缩或布局变化时，这两个按钮会跟随右侧头部一起发生挤压，出现排布异常
- 修改内容：
  - 将“设置”“更多”及其下拉菜单结构从右侧面板头部中剥离
  - 提升为页面级独立悬浮操作区
  - 保留原有按钮 `id` 与菜单结构，尽量不改动现有事件逻辑
  - 新结构不再参与左侧或右侧面板内部布局计算
- 当前行为：
  - 除“完全穿透”状态外，右上角这两个按钮始终固定显示
  - 进入“完全穿透”时，两个按钮与其菜单一并隐藏
- 影响范围：
  - 仅调整顶部两个全局操作按钮的挂载层级与显示策略
  - 不改变设置抽屉、更多菜单的既有功能逻辑
- 相关文件：
  - `public/index.html`
  - `public/styles.css`
  - `public/ui.shell.css`

### 3. AI 镜像映射控制独立浮层化 + 顶部全局控件窗口形状修正

- 当前行为：

  - `设置 / 更多` 与 `映射控制` 都独立于左右两个界面之上

### 4. 顶部控件分层职责修正

- 类型：Bug 修复 / 分层重构修正
- 原始问题：
  - 上一轮将“映射控制”也提升为全局浮层，偏离了真实需求
  - “设置 / 更多”按钮虽然提升到了全局层，但“更多菜单”本体没有完整切到独立显示层与窗口 shape 同步中
  - 导致右侧界面关闭、拉伸、AI 镜像嵌入切换时，菜单显示与交互仍存在异常
- 修改内容：

  - 保留其与右侧界面同层，但补充最大宽度与布局兼容，降低拉伸时挤压重叠问题
  - “设置 / 更多”按钮继续保留在左右界面之外的全局独立层
  - “更多菜单”改为固定定位弹层，由运行时根据按钮实际位置动态计算弹出坐标
  - 将“更多菜单”打开状态纳入 Electron 窗口 `shape` 计算，避免显示出来但点击区域未同步的问题
  - 在窗口 `resize / scroll / 布局变化` 时同步刷新菜单与映射控制位置
- 当前行为：
  - “映射控制”回归右侧界面，不再与“设置 / 更多”共层
  - “设置 / 更多”的菜单不会再因左右界面拉伸、关闭而被带偏
  - AI 镜像嵌入时，顶部控件显示层与交互层分离更稳定

### 8. 任务包1兼容性修复：正式界面无法打开

- 类型：`严重 Bug 修复 / 启动兼容修正`
- 现象：
  - 任务包1接入后，部分已有本地数据环境启动时会出现“正式界面打不开”的严重问题
  - 实际表现是启动页后主界面处于不可用状态，本质上是左右主面板被历史布局数据错误折叠，导致可见主界面为空
- 原因：
  - 任务包1首版把旧版 `left/right panel` 的持久化数据直接迁移到了新的左右主面板布局模型
  - 但旧版 `leftPanelCollapsed/leftPanelWidth` 的语义并不等于新方案里的“左侧主界面（画布）收起/宽度”
  - 历史用户数据中一旦存在旧的收起状态，启动时就可能把新的左右主界面一起迁移成不可见状态
- 修改内容：
  - 调整布局迁移逻辑，旧数据迁移时只保留安全信息，不再直接继承旧版左右收起状态
  - 增加布局可用性兜底校验：若读取到的布局会让左右主界面同时不可见，则自动回退到默认布局
  - 保留新的 `panelLayout` 持久化链路，不影响任务包1后续继续扩展
- 影响范围：
  - 修复已有本地用户数据环境下的启动兼容问题
  - 不改变当前任务包1已经接入的左右换位与布局底座结构
- 相关文件：
  - [public/src/runtime/workbenchRuntime.js](/d:/FreeFlow-WorkBoard/public/src/runtime/workbenchRuntime.js)

补充定位与修正：

- 上述兼容迁移问题修完后，继续排查发现启动链路还有一个更底层的问题：
  - `bootstrap()` 在 `workbenchRuntime.js` 中被过早调用
  - 启动过程中会进入 `setCanvasMode()`，而该函数内部会调用 `scheduleCanvasModeSliderSync()`
  - 但 `canvasModeSliderFrame` 的顶层声明位于文件更后方，模块尚未执行到该位置时就会触发 `Cannot access before initialization`
- 最终修正：
  - 移除中段的提前 `bootstrap()` 调用
  - 改为在模块尾部、所有顶层变量和事件注册完成后再启动
  - 增加统一的 `bootstrap` 失败兜底，防止再次出现异常后界面永久卡在启动页而没有任何提示
- 这次属于启动顺序错误修复，优先级高于布局兼容问题。

继续排查结果：

- 使用 Chromium 对 `http://127.0.0.1:3000/?desktop=1` 做页面级检查后，进一步发现任务包1还引入了一个启动级引用错误：
  - `workbenchRuntime.js` 在创建 `createConversationAssistantChrome(...)` 时传入了 `conversationShellSwapBtn`
  - 但顶部从 `getConversationAssistantElements()` 的解构中漏掉了这个变量
  - 导致页面启动时直接抛出 `ReferenceError: conversationShellSwapBtn is not defined`
  - 模块初始化被中断，`app-booting` 无法移除，因此界面持续卡在启动页
- 最终补修：
  - 将 `conversationShellSwapBtn` 正式补回 `workbenchRuntime.js` 的 DOM 元素解构列表
- Chromium 复测结果：
  - `app-booting = false`
  - 启动画面已隐藏
  - `page-shell` 已正常可见
  - 页面无新的 `pageerror`

### 9. 任务包2：主面板自由布局

- 类型：`架构调整 / 交互改造`
- 修改内容：

  - 将原有多条 `resize` 监听合并为单条 RAF 调度
  - 刷新时只使用当前内存中的 `panelLayout`，不再回读旧存储宽度
  - 统一在同一轮调度内触发：
    - 布局重排
    - shape 同步
    - 右侧滑块刷新
    - 画布模式滑块刷新
    - 映射工具栏刷新
- 影响范围：

  - 如果只是右键单击或短按未拖动，则保留原有右键菜单打开逻辑
- 影响范围：
  - 空白区右键单击重新恢复为打开画布二级菜单
  - 空白区右键长按并拖动仍然可以移动画布
