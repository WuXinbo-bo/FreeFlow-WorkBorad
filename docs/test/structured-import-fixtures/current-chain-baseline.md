# 当前链路行为快照

## 1. 目的

本文件用于记录 `TP-00 / WP-02` 阶段下，当前系统在复制 / 粘贴 / 拖拽导入上的核心行为基线。

该文件只描述“现在是什么”，不描述“应该改成什么”。

## 2. 当前关键代码入口

- 内部复制：
  - `public/src/engines/canvas2d-core/brokers/clipboardBroker.js`
- 外部导入：
  - `public/src/engines/canvas2d-core/brokers/dragBroker.js`
- 粘贴主入口：
  - `public/src/engines/canvas2d-core/createCanvas2DEngine.js`
- 富文本编辑器内部粘贴：
  - `public/src/engines/canvas2d-core/editors/richTextAdapter.js`

## 3. 当前行为摘要

### 3.1 内部复制

当前内部复制使用 `clipboardBroker.buildPayloadFromItems()` 构建 payload。

当前 payload 特征：

- 以 `items` 为核心，直接克隆原生画布元素
- 文本类元素会额外导出 `text` 与 `html`
- 图片 / 文件卡片会额外导出 `filePaths`

当前结论：

- 内部复制更偏“原生元素搬运”
- 不是统一结构化文档协议

### 3.2 外部 HTML 导入

当前 `dragBroker.importFromDataTransfer()` 对 `text/html` 的处理是：

1. 归一化 HTML 字号
2. 从 HTML 计算 plain text
3. 直接调用 `createTextElement(point, plainText, html)`

当前结论：

- HTML 不会先被解析成 block / inline 结构树
- 绝大部分 HTML 内容最终进入单个 `text` 元素

### 3.3 外部纯文本导入

当前 `text/plain` 会：

- 经 `sanitizeText`
- 直接落为单个 `text` 元素

当前结论：

- 纯文本链路简单稳定
- 但没有 paragraph 级语义表达

### 3.4 文件拖拽导入

当前文件拖拽会优先识别：

- 图片文件 -> 图片元素
- 非图片文件 -> 文件卡片元素

当前结论：

- 文件拖拽是当前链路里最“结构清晰”的一支
- 但与文本 / HTML 导入并不共享统一中间协议

### 3.5 图片与文件卡片体系

当前图片与文件体系已经拥有相对稳定的原生落地点：

- 图片 -> `image` 元素
- 文件 -> `fileCard` 元素

当前结论：

- 它们属于后续新体系必须纳入统一管理的对象
- 但不需要在第一阶段推翻已有元素模型
- 更合理的是先让它们进入统一输入、协议、复制与诊断链路

### 3.6 旧节点体系

当前 `flow` 节点与节点文本体系依然是现有画布的重要组成部分。

当前结论：

- 节点体系必须进入样本观察和回归范围
- 但第一阶段不建议强制升级为 canonical 原生节点
- 更合理的是先通过适配层与新结构化内容共存

### 3.7 图形、连线、手绘体系

当前几何图形、连线、手绘等元素更偏“画布原生创作元素”。

当前结论：

- 第一阶段不建议纳入结构化导入主线
- 但必须纳入复制、导出、选中兼容观察

### 3.8 右键粘贴

当前右键粘贴能力弱于原生 Ctrl + V：

- 主要依赖系统剪贴板文本和文件路径
- 不等价于完整的浏览器 `clipboardData` 链路

当前结论：

- 右键粘贴与 Ctrl + V 存在能力差异
- 这是后续统一 Import Gateway 的直接原因之一

### 3.9 命中与选中系统

当前命中逻辑主要服务于旧元素模型，重点围绕 `text`、`shape`、`flowEdge`、`fileCard`、`image` 等对象。

当前结论：

- 命中系统必须纳入后续新体系的独立观察范围
- 表格单元格、公式编辑区、代码块局部区域、task checkbox 等都不能只依赖整体 bounds 命中

### 3.10 搜索系统

当前搜索索引主要覆盖 `text`、`flowNode`、`fileCard`、`image` 等已有对象。

当前结论：

- 未来 `codeBlock`、`table`、`math`、`taskList`、`footnote` 等新结构默认不会自动进入搜索结果
- 搜索系统必须在后续独立新链路中预留结构化映射能力

### 3.11 导出链路

当前导出准备逻辑对 `text` 元素存在专门规范化处理，新元素尚无对应导出协议。

当前结论：

- 新结构化元素后续必须同步定义导出测量、降级与兼容输出策略
- 导入成功不等于导出可用，导出链路必须单独纳入观察

### 3.12 历史记录、状态与渲染注册

当前历史、store、renderer 注册机制都建立在旧元素体系之上。

当前结论：

- 表格编辑、任务勾选、公式源码切换、代码块语言切换等细粒度动作会给撤销重做带来新压力
- 独立新系统后续必须明确状态边界，并通过适配层与旧画布桥接
- 新 renderer 不应直接继续堆入旧 renderer 分支

### 3.13 工具体系与附属编辑区模式

当前选择工具、平移工具、创建工具以及文件卡片 / 图片的附属编辑区模式，已经构成画布交互的既有基础。

当前结论：

- 新结构化元素不只是 renderer 问题，也会影响工具状态和编辑态切换
- 主体元素 + 局部交互区的模式后续可复用，但必须统一协议

## 4. 当前链路的主要局限

- HTML / Markdown / Office / 代码 / 公式没有统一 parser
- `text` 元素承担了过多复杂结构
- 右键粘贴、拖拽、Ctrl + V 行为不统一
- 内部复制与外部导入不共享同一结构协议
- 图片、文件、节点等既有体系尚未进入统一导入与诊断管理框架
- 命中、搜索、导出、历史、状态、renderer、工具、附属编辑区等画布级系统尚未进入统一基线与设计框架

## 5. 本文件更新规则

- 每次发现旧链路关键行为与本文件不一致时，应同步修订
- 新旧链路切换阶段，本文件只用于描述“旧链路基线”
